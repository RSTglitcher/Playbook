:::tip Definition
**Project Architecture** describes how a software project is structured across code, dependencies, tooling, workflows, and team conventions — the system that shapes how engineers build, test, deploy, and operate software.
:::

**Typical examples include:**
- A Python service with `src/`, `tests/`, Poetry, Docker, and GitHub Actions  
- A JVM monolith with Maven, layered modules, and internal libraries  
- A data engineering repo with ETL folders, schemas, and pipeline configs  

---

## 🎯 What Problem Does This Solve?

Understanding **Project Architecture** enables:

| Benefit | Why it matters |
|--------|----------------|
| **Clarity of constraints** | Reveals what the team *must* keep vs what can change |
| **Template alignment** | Ensures cookiecutter templates support real-world needs |
| **Risk reduction** | Identifies brittle or non-standard structures early |
| **Faster onboarding** | New engineers understand the project shape quickly |
| **Better requirements** | Requirements become grounded in engineering reality |

---

## 🧠 Conceptual Model (How It Works)

A project’s architecture is shaped by:

### **Language & Runtime Constraints**
- Packaging model  
- Directory conventions  
- Build tools  
- Dependency resolution  

### **Team Workflow**
- CI/CD pipelines  
- Testing strategy  
- Deployment model  
- Observability patterns  

### **Domain Requirements**
- Data pipelines  
- API boundaries  
- Infrastructure-as-code  
- ML workflows  

### **Organisational Standards**
- Templates  
- Security requirements  
- Naming conventions  
- Internal libraries  

Together, these form the **project skeleton** — the structure that governs how work gets done.

---

## 🧭 When to Use

### **Use When**
- Designing or updating a project template  
- Reviewing a team’s repo to understand constraints  
- Identifying fixed vs flexible vs team‑specific elements  
- Diagnosing structural issues (build failures, dependency drift, CI pain)  

:::warning Avoid When
- Assuming all teams should follow identical structures  
- Forcing a template onto a team with domain‑specific needs  
- Treating flexible elements as fixed (or vice versa)  
:::

---

## 🔍 TA Lens

:::info How a TA Evaluates Project Architecture
- Identify **fixed elements** dictated by language, runtime, or organisation  
- Identify **flexible elements** that teams can change with effort  
- Identify **team‑specific elements** that must be supported via extension points  
- Look for structural signals: modularity, testability, dependency hygiene  
- Ask whether the architecture supports the team’s domain and workflow  
:::

## 📘 Key Terminology

| Term | Definition |
|------|------------|
| **Fixed Element** | A project component that cannot change without major cost |
| **Flexible Element** | A component that can change with effort and coordination |
| **Team‑Specific Element** | A domain‑driven component unique to a team’s workflow |
| **Extension Point** | A template mechanism allowing teams to customise behaviour |
| **Project Skeleton** | The baseline directory and tooling structure |

# 🧬 Variants / Types of Project Architecture

<Tabs>

<TabItem value="baseline" label="Baseline Structure">

### Baseline Project Structure

| Area | Components |
|------|------------|
| **Source Code** | Modules, packages, classes, functions, config files |
| **Dependencies** | Language deps, system deps, internal libraries |
| **Build & Packaging** | Build tool, package manager, artifact output |
| **Testing** | Unit tests, integration tests, test data |
| **CI/CD** | Linting, build, test, deploy |
| **Runtime & Deployment** | Container/VM/serverless, env config, secrets |
| **Observability** | Logging, metrics, health checks |

</TabItem>

<TabItem value="fixed" label="Fixed Elements">

### 🔒 Fixed Elements  
These are dictated by the language, runtime, or organisational constraints.

| Language | Typical Fixed Elements | Why Fixed |
|---------|-------------------------|-----------|
| **Java** | Maven/Gradle, `src/main/java`, JAR/WAR output | JVM conventions + ecosystem |
| **Scala** | sbt, multi‑module builds | Compiler + community norms |
| **Python** | `src/` layout, virtualenv/poetry | Packaging + interpreter behaviour |
| **Node.js** | `package.json`, npm/yarn | Runtime + dependency model |

**TA Interpretation:**  
Fixed elements are **non‑negotiable** and must be supported by templates.

</TabItem>

<TabItem value="flexible" label="Flexible Elements">

### 🔄 Flexible Elements  
These can change, but only with cost and coordination.

| Area | Typical Flexibility | TA Interpretation |
|------|----------------------|-------------------|
| Directory structure | Medium | Reorganisation is disruptive |
| Testing framework | Medium | Migration possible but painful |
| CI pipeline | Medium | Adaptable via extension points |
| Dockerfile | Medium | Standardisable if base image fits |

**TA Interpretation:**  
Flexible elements reveal **negotiation space** and team history.

</TabItem>

<TabItem value="team" label="Team‑Specific Elements">

### 🎨 Team‑Specific Elements  
Unique to the team’s domain, workflow, or architecture.

| Team Type | Team‑Specific Needs |
|-----------|---------------------|
| Data Engineering | ETL folders, schema files, pipeline configs |
| Backend API | Routing conventions, middleware patterns |
| ML/AI | Model folders, notebooks, experiment tracking |
| Platform | Terraform modules, infra scripts |

**TA Interpretation:**  
Team‑specific elements require **extension points**, not standardisation.

</TabItem>

</Tabs>

# 🧩 Side‑by‑Side Project Examples

<Tabs>

<TabItem value="java" label="Java">

```mermaid
project/
├── src/
│   ├── main/java/
│   └── test/java/
├── pom.xml
└── target/
```

**Fixed:** Maven, directory structure  
**Flexible:** Test framework  
**Team‑specific:** Internal libraries, domain modules

</TabItem>

<TabItem value="python" label="Python">

```mermaid
project/
├── src/project_name/
├── tests/
├── pyproject.toml
└── requirements.txt
```

**Fixed:** Packaging model  
**Flexible:** Test runner, dependency tool  
**Team‑specific:** Data folders, scripts

</TabItem>

<TabItem value="node" label="Node.js">

```mermaid
project/
├── src/
├── test/
├── package.json
└── node_modules/
```

**Fixed:** package.json  
**Flexible:** Framework (Express/Fastify)  
**Team‑specific:** API structure, middleware

</TabItem>

<TabItem value="scala" label="Scala">

```mermaid
project/
├── src/
│   ├── main/scala/
│   └── test/scala/
├── build.sbt
├── project/
│   └── plugins.sbt
└── target/
```

**Fixed:** sbt, directory structure, compiler settings  
**Flexible:** Multi‑module layout, test framework  
**Team‑specific:** Functional libraries, domain boundaries

</TabItem>

## 🔌 How to Interact With It

**Access Pattern:**  
Clone repo → inspect structure → map constraints → compare to template.

**Operations / Interfaces:**  
- Build tools (Maven, sbt, Poetry, npm)  
- CI pipelines  
- Dockerfiles  
- Internal libraries  

**Interaction Model:**  
Direct repo analysis, interviews with engineers, comparison against standards.

---

## 📊 What Results Normally Look Like

| Result Type | Description |
|-------------|-------------|
| **Project map** | Clear breakdown of fixed/flexible/team‑specific elements |
| **Template requirements** | Structured list of what the template must support |
| **Risk assessment** | Identification of brittle or inconsistent structures |
| **Extension points** | Areas where teams need customisation |

:::note Special Considerations
Some teams evolve organically; historical decisions often explain structural quirks.
:::

---

## ⚠️ Common Misunderstandings / Anti‑Patterns

- Treating all teams as identical  
- Over‑standardising and removing necessary flexibility  
- Ignoring domain‑specific folder structures  
- Assuming CI/CD can be fully templated  
- Forcing a template onto a team without understanding constraints  

---

## 🏗️ Real‑World Examples

- A data team requires schema folders that don’t fit a backend template  
- A Python team uses Poetry but the template assumes pip/requirements.txt  
- A JVM team needs multi‑module builds that the template must support  
- A platform team requires Terraform modules that don’t belong in other repos
