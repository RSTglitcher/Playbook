:::tip Definition
Data Structures & Conditionals describe how software organises information and makes decisions, forming the foundation of performance, correctness, and system behaviour.
:::

**When to Use**

- Understanding performance characteristics  
- Evaluating system behaviour under load  
- Reviewing logic, branching, or data access patterns  
- Debugging slow or incorrect behaviour  
- Designing flows, pipelines, or decision rules  

**When Not to Use**

- When discussing storage systems (use Storage Theory instead)  
- When modelling domain entities (use Data Modelling)  
- When evaluating distributed behaviour (use Communication Patterns)  
- When logic belongs in state machines or workflow engines  

---

## 🎯 What Problem Does This Solve?

Data Structures & Conditionals solve the problem of **how systems organise information and choose actions**.  
They give Technical Analysts the mental models needed to:

- Predict performance and scaling behaviour  
- Understand why systems slow down or misbehave  
- Evaluate branching logic for correctness  
- Identify bottlenecks caused by repeated scans or poor structure  
- Communicate clearly with engineers using shared vocabulary  

---

## 🧠 Conceptual Model

Software behaviour emerges from two primitives:

### Core Components

#### **1. Data Structures**  
How data is *organised* and *accessed*.

#### **2. Conditionals**  
How decisions are *made* based on that data.

Together, they determine:

- What the system *knows*  
- How it *chooses*  
- How it *scales*  
- How it *responds* under load  

### Axes of Variation

- **Ordered vs Unordered** (lists vs sets)  
- **Indexed vs Keyed** (arrays vs maps)  
- **Linear vs Hierarchical** (lists vs trees)  
- **Deterministic vs Branching** (simple vs nested conditionals)  
- **Constant‑time vs Linear‑time** (O(1) vs O(n))  

---


## 🔍 TA Lens

:::info How a TA Evaluates This Concept
- Identify the **problem shape** and map it to the right structure  
- Look for repeated linear scans (O(n))  
- Evaluate branching logic for clarity and completeness  
- Understand how structure and logic behave under load  
- Surface risks around ordering, uniqueness, or lookup speed  
:::

**What happens when:**

- **Data grows** → linear scans become bottlenecks  
- **Traffic increases** → branching logic becomes hot paths  
- **Concurrency rises** → shared structures become contention points  
- **Resources become constrained** → inefficient structures degrade sharply

### Collections

#### Arrays / Lists

```
Index → 0  1  2  3
        [A][B][C][D]
```

**Purpose**  
Ordered, index‑based storage.

**Key Characteristics**
- Fast indexed access  
- Good for scans  
- Slow inserts/deletes  

**Behaviour**  
Performance degrades linearly as size grows.

**Trade-offs**  
Simple and predictable, but costly for mutation.

---

#### Sets

```
{ A, B, C }   // No duplicates
```

**Purpose**  
Uniqueness and fast membership checks.

**Key Characteristics**
- O(1) lookup  
- No ordering  

**Behaviour**  
Ideal for deduplication and “contains?” checks.

**Trade-offs**  
Hashing overhead; no positional access.

---

#### Maps / Dictionaries

```
Key → Value
A   → 1
B   → 2
```

**Purpose**  
Fast lookup by key.

**Key Characteristics**
- O(1) access  
- Natural modelling of relationships  

**Behaviour**  
Excellent for caching, indexing, and routing.

**Trade-offs**  
Memory overhead; collisions degrade performance.

---

#### Queues (FIFO)

```
Front → [A][B][C] → Back
```

**Purpose**  
Ordered processing.

**Use When**  
Task scheduling, message processing.

---

#### Stacks (LIFO)

```
Top → [C]
       [B]
       [A]
```

**Purpose**  
Last‑in, first‑out behaviour.

**Use When**  
Undo operations, DFS traversal.

### Trees

```
      A
     / \
    B   C
```

**Purpose**  
Hierarchical modelling.

**Key Characteristics**
- Fast search when balanced  
- Depth affects lookup  

**Behaviour**  
Used in indexing, routing, and file systems.

**Trade-offs**  
Unbalanced trees degrade to linear behaviour.

---

### Graphs

```
A — B — C
 \     /
   D
```

**Purpose**  
Model relationships and networks.

**Key Characteristics**
- Flexible structure  
- Supports cycles  

**Behaviour**  
Used in routing, dependencies, social graphs.

**Trade-offs**  
Traversal cost varies; cycles complicate logic.

</TabItem>

### Big‑O (TA Edition)

You don’t need formulas — you need **shapes**.

| Operation | Array/List | Set | Map |
|-----------|------------|-----|-----|
| Lookup | O(n) | O(1) | O(1) |
| Insert | O(n) | O(1) | O(1) |
| Delete | O(n) | O(1) | O(1) |

:::note TA Takeaway
Repeated linear scans (O(n)) are the root cause of many performance issues.
:::

<TabItem value="conditionals" label="Conditionals & Control Flow">

### Conditionals

#### IF / ELSE — Binary Decisions

**Purpose**  
Choose between two behaviours.

**Key Characteristics**
- Simple branching  
- Clear fallback path  

**Behaviour**
```
WHEN <condition>
THEN <action>
ELSE <fallback>
```

---

#### Multi‑branch (IF / ELIF / ELSE)

**Purpose**  
Choose between multiple paths.

**Behaviour**
```
WHEN <A> → <action A>
WHEN <B> → <action B>
ELSE → <fallback>
```

---

#### CASE / SWITCH — Category Decisions

**Purpose**  
Match categories or types.

**Behaviour**
```
WHEN <category1> → <action1>
WHEN <category2> → <action2>
OTHERWISE → <fallback>
```

---

#### Loops — Repeated Conditional Behaviour

**FOR** → known iteration count  
**WHILE** → repeat while condition is true  
**UNTIL** → repeat until condition becomes true  

---

#### Functions — Encapsulated Logic

**Purpose**  
Group conditional logic into reusable units.

**Behaviour**
```
DEFINE function:
    WHEN <condition>
    THEN <action>
    ELSE <fallback>
```

## 🧩 System Interactions

:::info How a TA Understands the System
- How data structures interact with memory, CPU, and storage  
- How branching logic interacts with runtime and concurrency  
- What becomes a bottleneck as data or traffic grows  
:::

### Local Systems

- Memory layout  
- CPU cache behaviour  
- Runtime data access  
- Concurrency and locking  
- Sorting and traversal  

### Remote Systems

- Distributed graphs  
- Query engines  
- Caches and indexes  
- Message queues  

### Questions to ask during reviews or incidents

- What structure is being used, and is it appropriate?  
- Are we scanning repeatedly?  
- Are conditions complete and non‑contradictory?  
- What happens when data doubles?  
- Are we missing edge cases?

## 💥 Outputs / Results

:::note Special Considerations
Performance issues often arise from structure/logic mismatches rather than code bugs.
:::

### Success Modes

| Result Type | Description |
|-------------|-------------|
| Predictable Performance | Operations scale as expected |
| Clear Decision Logic | Branching is explicit and correct |
| Efficient Access Patterns | Minimal scanning or unnecessary work |
| Stable Behaviour Under Load | No sudden degradation |

### Failure Modes

| Failure Type | Description |
|--------------|-------------|
| Linear Scan Bottlenecks | Repeated O(n) operations |
| Incorrect Branching | Missing or contradictory conditions |
| Explosive Traversal | Graph cycles or deep recursion |
| Unnecessary Sorting | Wasted computation in pipelines |
