:::tip Definition
**Code Security** refers to the patterns, principles, and safeguards that prevent vulnerabilities in software systems.
It focuses on protecting data, preventing misuse, and ensuring that code behaves safely under malicious or unexpected conditions.
:::

Code security is not about tools — it is about **recognising risk patterns**, **understanding attack surfaces**, and **ensuring safe defaults**.

---

## 🎯 What Problem Does Code Security Solve?

Code Security enables:

| Benefit | Why it matters |
|--------|----------------|
| **Protection of data** | Prevents leaks, corruption, and unauthorised access |
| **Resilience to attacks** | Reduces risk of exploitation and downtime |
| **Predictable behaviour** | Prevents unexpected or unsafe execution paths |
| **Trustworthiness** | Ensures systems behave safely under adversarial input |
| **Compliance** | Supports regulatory and organisational requirements |

---

## 🧠 Conceptual Model (How Code Security Works)

Security issues arise when **untrusted input** interacts with:

- code
- data stores
- interpreters
- file systems
- networks
- authentication systems

Code Security patterns focus on:

### **1. Input Validation**
Ensure data is safe before use.

### **2. Output Encoding**
Ensure data is safe before display or transmission.

### **3. Authentication & Authorisation**
Ensure only the right people or systems can act.

### **4. Data Protection**
Ensure sensitive data is stored and transmitted safely.

### **5. Safe Defaults**
Ensure systems fail closed, not open.

### **6. Dependency & Supply Chain Safety**
Ensure third‑party code is trustworthy.

# 🧬 Core Code Security Patterns

<Tabs>

<TabItem value="input" label="Input Validation Pattern">

### Purpose
Prevent malicious or malformed input from being executed or trusted.

### Characteristics
- Validate type, length, format
- Reject unexpected characters
- Use allowlists over denylists

### Problem Shapes
- User input
- Query parameters
- File uploads
- API payloads

### TA Lens
Ask: *“What input is untrusted, and how is it validated?”*

</TabItem>

<TabItem value="output" label="Output Encoding Pattern">

### Purpose
Prevent untrusted data from being interpreted as code.

### Characteristics
- Escape HTML, SQL, shell, or JSON contexts
- Encode before rendering
- Prevent injection attacks

### Problem Shapes
- UI rendering
- Logs
- Dynamic templates

### TA Lens
Ask: *“Where is untrusted data being output, and in what context?”*

</TabItem>

<TabItem value="auth" label="Authentication & Authorisation Pattern">

### Purpose
Ensure only legitimate users or systems can act.

### Characteristics
- Strong authentication
- Role‑based or attribute‑based access
- Least privilege
- Session management

### Problem Shapes
- API access
- Admin interfaces
- Multi‑tenant systems

### TA Lens
Ask: *“Who is allowed to do this, and how is that enforced?”*

</TabItem>

<TabItem value="data" label="Data Protection Pattern">

### Purpose
Protect sensitive data at rest and in transit.

### Characteristics
- Encryption
- Hashing
- Key management
- Secrets isolation

### Problem Shapes
- Credentials
- Personal data
- Tokens
- Payment information

### TA Lens
Ask: *“What data is sensitive, and how is it protected?”*

</TabItem>

<TabItem value="error" label="Error & Exception Safety Pattern">

### Purpose
Prevent information leakage and unsafe fallback behaviour.

### Characteristics
- Generic error messages
- No stack traces to users
- Fail closed, not open

### Problem Shapes
- Authentication failures
- API errors
- Validation failures

### TA Lens
Ask: *“What happens when this fails, and is that safe?”*

</TabItem>

<TabItem value="dependency" label="Dependency & Supply Chain Pattern">

### Purpose
Prevent vulnerabilities introduced by third‑party code.

### Characteristics
- Version pinning
- Vulnerability scanning
- Minimal dependencies
- Trusted sources

### Problem Shapes
- Package managers
- Open‑source libraries
- Build pipelines

### TA Lens
Ask: *“What external code do we rely on, and is it safe?”*

</TabItem>

</Tabs>

# 🧩 Common Vulnerability Patterns (TA‑Friendly)

These are the **problem shapes** behind most real‑world vulnerabilities.

### **Injection Pattern**
Untrusted input becomes executable code.
Examples: SQL injection, command injection, template injection.

### **Broken Access Control Pattern**
Users can access data or actions they shouldn’t.

### **Authentication Weakness Pattern**
Weak or missing authentication.

### **Insecure Deserialization Pattern**
Untrusted data becomes objects or code.

### **Misconfiguration Pattern**
Unsafe defaults, exposed admin panels, debug modes.

### **Sensitive Data Exposure Pattern**
Unencrypted data, logs containing secrets.

### **Cross‑Site Scripting (XSS) Pattern**
Untrusted data rendered as HTML/JS.

### **Cross‑Site Request Forgery (CSRF) Pattern**
Unwanted actions triggered by another site.

### **Dependency Vulnerability Pattern**
Outdated or compromised libraries.

# 🧭 When to Use Code Security Patterns

Use these patterns when:

- Reviewing API or service designs
- Evaluating data flows
- Assessing risk in new features
- Reviewing pull requests
- Working with external integrations
- Handling sensitive data

---

# 🚫 When Not to Use

Avoid over‑security when:

- The system is isolated and non‑critical
- The cost of controls outweighs the risk
- You are still in early prototyping

Security should be **proportionate**, not maximal.

# 📘 Key Terminology

| Term | Definition |
|------|------------|
| **Attack Surface** | All points where an attacker can interact with the system |
| **Threat Model** | Assumptions about what attackers can do |
| **Least Privilege** | Give only the permissions required |
| **Secure Defaults** | Systems should be safe without configuration |
| **Hardening** | Reducing attack surface |
| **Secret** | Sensitive value requiring protection |

###  ⚠️ Common Mistakes

- “Security is an engineering problem”
- “We’ll fix it later”
- “It’s internal, so it’s safe”
- “We trust our users”
- “Validation is optional”
- “Dependencies are safe by default”
