:::tip Definition
**File Types & Data Formats** describe how information is encoded, structured, and represented for storage, transmission, configuration, or processing.  
They define *how data looks*, *how it is parsed*, and *how systems interpret it*.
:::

File types are not storage systems — they are **formats** that shape interoperability, performance, and correctness.

---

## 🎯 What Problem Does This Solve?

Understanding file types enables:

| Benefit | Why it matters |
|--------|----------------|
| **Correct parsing** | Prevents misinterpretation between systems |
| **Interoperability** | Enables communication across languages, platforms, and services |
| **Performance optimisation** | Choose formats suited for analytics, messaging, or configuration |
| **Schema enforcement** | Ensures data integrity and predictable structure |
| **Debuggability** | Human‑readable formats simplify troubleshooting |

---

## 🧠 Conceptual Model (How File Types Work)

File types vary along several axes:

### Text vs Binary
- **Text** → human‑readable, easy to debug  
- **Binary** → compact, fast, schema‑driven  

### Schema‑less vs Schema‑enforced
- JSON, CSV → flexible but error‑prone  
- Protobuf, Avro, XSD → strict structure, safer evolution  

### Row‑based vs Columnar
- CSV → row‑oriented  
- Parquet → column‑oriented for analytics  

### Configuration vs Data vs Code
- YAML/TOML → configuration  
- JSON/XML → structured data  
- SQL/GraphQL → schema or query definitions

**Purpose:**
Encode structured data for storage or transmission.

### JSON (`.json`)
Human‑readable key/value structured data.

**Use When**
- REST APIs  
- Configuration  
- Logging/debugging  
- Cross‑language communication  

**Characteristics**
- Lightweight  
- Flexible  
- No enforced schema  
- Text-based format
- Key/value pairs
- Supports numbers, strings, booleans, arrays, null
- Ignores whitespace
- Easy to parse and generate

---

### XML (`.xml`)
Markup‑based structured data with optional schemas.

**Use When**
- Enterprise integrations  
- Document‑centric data  
- Systems requiring strict validation
- Use XML when strict validation, namespaces, or enterprise compatibility are required.

**Characteristics**
- Verbose  
- Supports namespaces  
- Often paired with XSD
- Text-based format
- Requires closing tags
- Supports nested elements
- Often paired with XSD for validation
- Harder to read than JSON

---
### Protobuf (`.proto`)
Compact, schema‑driven binary format.

**Use When**
- High‑performance services  
- gRPC  
- Low‑latency communication  

**Characteristics**
- Strong schema  
- Very small payloads  
- Fast serialization
- Binary-based format

---

### Avro (`.avsc`)
Schema‑based binary format used in data pipelines.

**Use When**
- Event streaming  
- Schema evolution  
- Kafka ecosystems  

**Characteristics**
- Self‑describing  
- Good for long‑term storage  
- Efficient for ingestion pipelines
- Binary-based format

## 🧭 When to Use Which Format

| Need | Best Fit |
|------|----------|
| Human‑readable structured data | JSON, YAML |
| High‑performance messaging | Protobuf |
| Schema evolution in pipelines | Avro |
| Analytics at scale | Parquet |
| Simple tabular exchange | CSV |
| Strict document validation | XML + XSD |
| Configuration | YAML, TOML, ENV |

## 🧠 TA Lens

:::info How a TA Evaluates File Types
- What guarantees are required (schema, ordering, types)?  
- Is human readability important?  
- Is performance or payload size critical?  
- Will the format evolve over time?  
- Does the system require strict validation?  
- Is the data for operational use or analytics?  
:::

###  ⚠️ Common Mistakes

- Using CSV for complex nested data
- Using JSON when strict schemas are required
- Using YAML for secrets (indentation errors, parsing issues)
- Using XML without XSD in regulated environments
- Using Parquet for operational workloads
- Treating file formats as storage systems

## 🔗 Related Runbook Concepts

- BigQuery / Snowflake / Redshift — analytical engines that read columnar formats such as Parquet and ORC
- Databricks / Spark / Flink — distributed compute engines that process Parquet, Avro, and other analytical formats
- Kafka / Pub/Sub — messaging systems that commonly use Avro or Protobuf schemas for event payloads
- Schema Registries — systems that manage Avro/Protobuf schemas and enforce compatibility
- GCS / S3 Data Lakes — object stores that hold Parquet, ORC, Avro, and CSV files for analytical workloads
- Airflow / Cloud Composer — ETL/ELT orchestration systems that transform and materialise data into specific file formats
- API Gateways / gRPC Services — operational systems that emit JSON, XML, or Protobuf payloads
- Metadata & Governance Systems (Data Catalog, Amundsen) — tools that track schema definitions, file formats, and lineage
