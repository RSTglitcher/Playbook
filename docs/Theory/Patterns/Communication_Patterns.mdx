:::tip Definition
Communication Patterns describe the structured ways in which software components exchange information.
They define *when*, *how*, and *under what guarantees* data moves between services.
:::

### When to Use

- **Synchronous APIs** — immediate result required; caller must block.
- **Asynchronous Messaging** — decoupling, background work, retries, or delayed processing.
- **Event Streams** — high‑throughput, ordered, replayable data with multiple consumers.
- **Protocols** — real‑time, bidirectional, low‑latency communication (IoT, mobile, embedded).
- **File‑based / Batch** — partner or legacy systems without API/messaging endpoints.


### When Not to Use

- **Hidden domain boundaries** — communication used to mask unclear ownership.
- **Shared database as integration** — implicit coupling and brittle contracts.
- **Unclear or mismatched latency needs** — sync fails when slow; async fails when immediate.
- **Incorrect delivery guarantees** — semantics don’t match business requirements.
- **Misunderstood ordering/replay needs** — choosing messaging when streams are required (or vice versa).
- **Insufficient operational maturity** — lacking observability, retries, DLQs, or backpressure handling.


---

## 🎯 What Problem Does This Solve?

Communication Patterns exist to solve the fundamental problem of how independent services in a distributed system
exchange information reliably and predictably.

They give us a structured way to decide when communication should be synchronous or asynchronous, what delivery and ordering
guarantees are required, and how failures, retries, and backpressure should be handled.

As a Technical Analyst, understanding these patterns is essential because they **directly shape system behaviour** under
load, during failures, and as the architecture evolves. Choosing the right pattern **prevents common issues** like latency
spikes, inconsistent state, tight coupling, and brittle integrations — and choosing the wrong one is one of the fastest ways
to create systemic fragility.

## 🧠 Conceptual Model

### Core Components

- **Producer / Client** — initiates communication (request, message, event).
- **Consumer / Server** — receives and processes the communication.
- **Transport** — the channel (HTTP, TCP, WebSockets, MQTT, file transfer).
- **Broker / Log (optional)** — intermediaries that store, buffer, or route messages.
- **Contract** — schema or API definition that governs compatibility.

### Axes of Variation

#### Synchronous vs Asynchronous
- Synchronous → request/response, immediate result
- Asynchronous → messages, events, queues, streams

#### Push vs Pull
- Push → sender initiates delivery
- Pull → receiver fetches data

#### Transport vs Semantics
- Transport → TCP, HTTP, WebSockets
- Semantics → REST, GraphQL, RPC, event schemas

#### Point‑to‑point vs Broadcast
- Direct communication
- Fan‑out to multiple consumers

#### Durability vs Latency
- Durable queues vs low‑latency protocols

---

### Typical Lifecycle or Flow

A generic communication lifecycle looks like:

1. **Create** — producer forms a request/message/event.
2. **Encode** — data is serialized according to the contract.
3. **Transmit** — transport delivers the data (push or pull).
4. **Process** — consumer handles the data (sync or async).
5. **Acknowledge** — success/failure is recorded (response, ack, offset).
6. **Retry / Replay (optional)** — based on guarantees and failure mode.
7. **Observe** — logs, metrics, and traces capture behaviour.

#### REST

<details>
<summary>Mental Map</summary>

```mermaid
Client --> HTTP Request --> REST API --> Business Logic --> Data Store
Client <-- HTTP Response <-------------------------------/

|-Client — initiates the request and waits for a response
|- |- HTTP Request — carries method, headers, and payload
|- |- |- REST API — exposes resources via standard verbs
|- |- |- |- Business Logic — applies rules, validation, orchestration
|- |- |- |- |- Data Store — persistent system of record
|- |- |- |- |- |- HTTP Response — structured representation of the resource
```

How It Connects Together:
- The client blocks until the server responds
- Each request is stateless and independent

</details>

#### GraphQL

<details>
<summary>Mental Map</summary>

```
mermaid
Client --> GraphQL Query --> GraphQL Server
                     |--> Resolver A --> Service A
                     |--> Resolver B --> Service B
                     |--> Resolver C --> Service C
Client <-- GraphQL Response (exact requested shape)

|- GraphQL Query — client specifies the exact shape of data
|- |- GraphQL Server — validates query and orchestrates resolvers
|- |- |- Resolvers — fetch data from services or databases
|- |- |- |- Services — provide domain‑specific data
|- |- |- |- |- GraphQL Response — matches the client’s requested shape
```

How It Connects Together:
- GraphQL centralises orchestration but decentralises data ownership
- Clients avoid over‑fetching and under‑fetching
- Resolver chains can create synchronous dependency waterfalls
- Strong schema and introspection support tooling

</details>

#### RPC / gRPC

<details>
<summary>Mental Map</summary>

```
mermaid
Client --> RPC Call --> gRPC Server --> Handler --> Data/Services
Client <-- Binary Response <-------------------------------/

|- RPC Call — function‑like invocation over the network
|- |- gRPC Server — exposes strongly typed service methods
|- |- |- Handler — executes the remote function
|- |- |- |- Data/Services — underlying systems providing results
|- |- |- |- |- Binary Response — efficient protobuf‑encoded payload
```

How It Connects Together:
- RPC feels like calling a local function, but is synchronous
- Strong contracts enforce compatibility
- High performance, low latency
- Tight coupling between client and server versions

</details>

#### Webhooks

<details>
<summary>Mental Map</summary>

```
mermaid
Producer Event --> HTTP POST --> Consumer Endpoint
Producer <-- 2xx ACK / Retry Logic --> Consumer
Concepts in the Flow

|- Producer Event — something happens that triggers a callback
|- |- HTTP POST — push‑based delivery
|- |- |- Consumer Endpoint — must be reachable and available
|- |- |- |- ACK / Retry Logic — ensures delivery attempts
```

How It Connects Together:
- Lightweight way to push notifications
- Delivery reliability depends on retries
- Consumers must expose public endpoints
- Good for simple event notifications

</details>

#### Message Queue

<details>
<summary>Mental Map</summary>

```
mermaid
Producer --> Queue --> Consumer Worker
Producer --> Queue --> Consumer Worker
Producer --> Queue --> Consumer Worker

|- Producer — sends individual work items
|- |- Queue — stores messages until processed
|- |- |- Consumer Worker — processes one message at a time
```

How It Connects Together:
- Each message is consumed by exactly one worker
- Great for background jobs and load‑levelling
- Ordering is not guaranteed
- Duplication and retries must be handled by consumers

</details>

#### Pub/Sub

<details>
<summary>Mental Map</summary>

```mermaid
Producer --> Topic --> Subscriber A
                 \--> Subscriber B
                 \--> Subscriber C

|- Producer — emits events without knowing who listens
|- |- Topic — broadcast channel
|- |- |- Subscribers — each receives its own copy
```

How It Connects Together:
- Enables fan‑out to multiple teams or systems
- Subscribers operate independently
- DLQs fill when consumers fall behind
- Schema drift becomes a risk across many consumers

</details>


#### Event Streams

<details>
<summary>Mental Map</summary>

```
mermaid
Service A --> Fragment A --> Topic
Service B --> Fragment B --> Topic
Service C --> Fragment C --> Topic

Topic --> Stream Processor --> Join + State + Window --> Assembled Object

|- Fragments — partial data emitted by different services
|- |- Events — immutable, timestamped records
|- |- |- Topics — durable logs storing events in order
|- |- |- |- Streams — events read over time at consumer pace
|- |- |- |- |- Join — correlation key aligns fragments
|- |- |- |- |- |- State — temporary storage for partial objects
|- |- |- |- |- |- |- Window — bounded waiting period
|- |- |- |- |- |- |- |- Assembled Object — first time the “whole” exists
```

How It Connects Together:
- **Join** - Correlation Key in common between fragments
- **State** - Temporary storage until all fragments arrive
- **Time Windows** - A set time period to wait for all fragments to arrive
- **Event-driven system contract**:
> - correlation ID
> - schema
> - meaning of each fragment
> - ownership
> - rules for evolution
> - expected behaviour of producers

</details>

#### HTTP/HTTPS

<details>
<summary>Mental Map</summary>

```
mermaid

Client --> HTTP Request --> Server --> Application Logic
Client <-- HTTP Response <-----------------------------/

|- HTTP Request — method, headers, body
|- |- Server — handles routing and processing
|- |- |- Application Logic — business rules
|- |- |- |- HTTP Response — structured output
```

How It Connects Together:
- Foundation for REST, GraphQL, Webhooks
- Request/response semantics
- Reliable, widely supported
- Latency and availability directly impact clients

</details>

#### TCP/UDP

<details>
<summary>Mental Map</summary>

```
mermaid
TCP:
Client <== Reliable, Ordered Stream ==> Server

UDP:
Client -- Unreliable, Unordered Packets --> Server

|- TCP — connection‑oriented, ordered, reliable
|- |- UDP — connectionless, fast, lossy
|- |- |- Packets/Frames — low‑level transport units
```

How It Connects Together:
- Forms the foundation for higher‑level protocols
- TCP guarantees delivery; UDP prioritises speed
- Applications must add semantics on top

</details>

#### WebSockets

<details>
<summary>Mental Map</summary>

```
mermaid
HTTP Upgrade --> WebSocket
Client <==================> Server
     (Full‑duplex, real‑time)

|- HTTP Upgrade — switches from HTTP to WebSocket
|- |- Full‑duplex Channel — both sides send anytime
|- |- |- Persistent Connection — long‑lived session
```

How It Connects Together:
- Ideal for chat, dashboards, multiplayer systems
- Requires connection management
- Stateful at scale

</details>

#### MQTT / CoAP

<details>
<summary>Mental Map</summary>

```
mermaid
MQTT:
Device --> Broker --> Subscribers

CoAP:
Device <--> CoAP Server (REST‑like over UDP)

|- MQTT Broker — central message router
|- |- Subscribers — receive published messages
|- |- |- CoAP Server — lightweight REST over UDP
|- |- |- |- Devices — constrained IoT clients
```

How It Connects Together:
- Designed for low‑power, unreliable networks
- MQTT uses pub/sub; CoAP uses request/response
- Lightweight, battery‑friendly, small payloads

</details>

## 🔍 TA Lens

:::info How a TA Evaluates This Concept
- What changes, what stays constant, and what becomes a bottleneck as the system evolves
- How the communication pattern behaves under pressure (latency, load, retries, backpressure)
- What guarantees it provides (ordering, durability, delivery semantics) and how they degrade
- What questions to ask during design reviews, incident analysis, or cross‑team integration
- What risks, constraints, or trade‑offs must be surfaced early to avoid systemic fragility
:::

### **What happens when…**

#### **Data grows**
- Does the pattern preserve ordering, durability, and replay semantics at scale?
- Do logs, queues, or buffers grow unbounded?
- Does schema evolution remain safe as message volume increases?

#### **Traffic increases**
- Do synchronous calls amplify tail latency?
- Do asynchronous queues absorb spikes or collapse into backlog?
- Does the broker/log become a hotspot or single point of contention?

#### **Concurrency rises**
- Are consumers able to scale horizontally?
- Does idempotency hold under parallel processing?
- Do locks, partitions, or shards introduce contention?

#### **Resources become constrained**
- Does backpressure activate?
- Do retries create cascading load?
- Does the system degrade gracefully or fail abruptly?

## 🧬 Variants / Types

<Tabs>
<TabItem value="apis" label="APIs (Synchronous)">

### APIs (Synchronous)

**Key Characteristics**
- Request/response
- Immediate feedback
- Coupled to availability and latency

**Core Patterns**
- **REST** — resource-based, stateless
- **GraphQL** — client-driven queries
- **RPC / gRPC** — function calls over the network
- **Webhooks** — server-to-server callbacks

**Behaviour**
- Latency-sensitive
- Cascading failures if dependencies are slow
- Requires retries, timeouts, and circuit breakers

**Trade-offs**
- Simple mental model
- Tight coupling
- Harder to scale under load

**Exceptions (Not Universal)**
- Some RPC systems support streaming
- Some APIs allow async callbacks

</TabItem>

<TabItem value="messaging" label="Messaging (Asynchronous)">

### Messaging (Asynchronous)

**Key Characteristics**
- Decoupled communication
- Durable delivery
- Absorbs traffic spikes

**Core Patterns**
- **Message Queues** — point-to-point work distribution
- **Pub/Sub** — broadcast to multiple consumers
- **Event Streams** — ordered, replayable logs

**Behaviour**
- Eventual consistency
- High throughput
- Requires idempotency and consumer design

**Trade-offs**
- Resilient and scalable
- Harder debugging and tracing

**Exceptions (Not Universal)**
- Some queues do not guarantee ordering
- Some streams do not guarantee exactly-once delivery

</TabItem>
<TabItem value="protocols" label="Protocols">

### Protocols

**Key Characteristics**
- Define how data is encoded and transported
- Influence latency, reliability, and ordering

**Core Protocols**
- **HTTP/HTTPS** — ubiquitous, request/response
- **TCP/UDP** — transport-level primitives
- **WebSockets** — bidirectional, real-time
- **MQTT / CoAP** — lightweight IoT protocols

**Behaviour**
- Transport determines reliability
- Protocol determines semantics

**Trade-offs**
- Flexibility vs complexity
- Reliability vs latency

**Exceptions (Not Universal)**
- UDP provides no delivery guarantees
- WebSockets require persistent connections

</TabItem>
<TabItem value="batch" label="File-Base">

### Batch Processing: SFTP / FTPS / SCP

**Key Characteristics**
- Moves files between systems over secure channels
- High latency; not real‑time
- No push semantics — consumers poll or schedule
- No delivery guarantees beyond “file successfully transferred”
- Operational overhead (keys, permissions, folder conventions)

**Use When**
- Partners cannot support APIs or messaging
- Large batch files must be exchanged
- Simplicity outweighs real‑time needs
- Regulatory or legacy constraints require file drops

**Trade‑offs**
- Simple and predictable
- Easy to audit
- High latency and no event semantics
- Requires strict naming conventions and folder contracts
- Failures often discovered only after batch completion

**Exceptions (Not Universal)**
- Some SFTP setups include checksum validation
- Some organisations wrap SFTP in orchestration tools for retries and monitoring


</TabItem>

<TabItem value="other" label="Other">

### Other Communication Methods

#### Shared Databases (Anti-Pattern)
- Tight coupling
- Hidden communication
- Breaks domain boundaries

#### File-Based Communication
- Batch-friendly
- High latency
- Simple integration

#### Pipes & Sockets
- Low-level, high-performance
- Requires careful coordination

#### Exceptions (Not Universal)
- Some legacy systems rely on shared DBs by necessity
- File-based communication is valid for batch ETL

## 🧩 System Interactions

:::info How a TA Understands the System
- How this concept interacts with architecture, data, and runtime environments
- How communication behaves under pressure (latency, load, retries, backpressure)
- What changes, what stays constant, and what becomes a bottleneck as the system evolves
:::

Communication patterns don’t operate in isolation — they sit at the intersection of compute, storage, network, and orchestration.
A TA evaluates them by understanding **where the pattern touches the system** and **how those touchpoints behave under stress**.

### Local Systems

How communication patterns interact with the local execution environment:

#### **OS**
- Thread scheduling affects synchronous latency
- File descriptors, sockets, and buffers limit concurrent connections
- Kernel networking stack influences throughput and packet handling

#### **Runtime**
- Serialization/deserialization cost
- Garbage collection pauses affecting latency
- Event loops, async runtimes, and concurrency models

#### **Network**
- Bandwidth, jitter, packet loss, MTU size
- TCP congestion control and retransmission
- Local network saturation causing cascading latency

#### **Storage**
- Local logs, buffers, and spill‑to‑disk behaviour
- Durability guarantees depend on fsync, disk I/O, and write‑ahead logs
- Backpressure when storage fills

#### **Concurrency**
- Thread pools, worker pools, async tasks
- Contention on locks, queues, or shared state
- Idempotency requirements under parallel processing

#### **Scaling**
- Horizontal scaling of consumers or API servers
- Local resource exhaustion (CPU, memory, file handles)
- How scaling interacts with ordering, partitioning, or sharding

---

### Remote Systems

How communication patterns interact with distributed infrastructure:

#### **Pods / Containers**
- Pod restarts cause dropped connections or in‑flight message loss
- Autoscaling changes concurrency and ordering behaviour
- Resource limits (CPU/memory) influence latency and throughput

#### **Data Centers / Regions**
- Cross‑region latency affects synchronous patterns dramatically
- Network partitions create divergent state in async systems
- Replication lag impacts ordering, replay, and consistency

#### **Service Mesh / Gateways**
- Retries, timeouts, and circuit breakers alter communication semantics
- mTLS, routing rules, and load balancing influence latency and availability

#### **Brokers / Logs**
- Kafka, RabbitMQ, SQS, Pub/Sub introduce their own durability, ordering, and backpressure behaviours
- Partitioning and consumer groups shape throughput and concurrency

## 💥 Outputs / Results

:::note Special Considerations
Communication patterns differ in how they serialize, deliver, and acknowledge data. Edge cases often arise from:
- schema evolution
- ordering guarantees
- retries and idempotency
- partial delivery or partial failure
:::

### Success Modes

What does success look like?

| Result Type | Description |
|-------------|-------------|
| **Timely Delivery** | Messages, requests, or events arrive within expected latency bounds; no backlog or consumer lag. |
| **Correct Ordering / Replay** | Event streams preserve ordering; consumers advance offsets smoothly without gaps. |
| **Stable Throughput** | System handles expected load without retries, throttling, or backpressure signals. |
| **Contract Compatibility** | Producers and consumers agree on schema/API; no deserialization or validation errors. |
| **Clear Acknowledgement** | ACKs, responses, or offset commits reflect successful processing. |

### Failure Modes

What symptoms in logs, metrics, or behaviour indicate failure?

| Failure Type | Description |
|--------------|-------------|
| **Latency Spikes** | Slow downstream dependencies cause synchronous calls to stall or async queues to grow. |
| **Retry Storms** | Repeated failures trigger exponential retries, increasing load and masking root causes. |
| **Inconsistent State** | Out‑of‑order messages, duplicate deliveries, or partial writes lead to divergence across services. |
| **Backpressure / Throttling** | Consumers cannot keep up; queue depth grows, or producers receive “slow down” signals. |
| **Schema / Contract Breakage** | Deserialization errors, incompatible field changes, or missing required fields. |
| **Silent Drops / DLQ Growth** | Messages are discarded, dead‑lettered, or never acknowledged causing duplicated events. |

## 📊 Output


| Pattern | Typical Output | **Evidence of Success** | **Evidence of Failure** |
|---------|----------------|--------------------------|---------------------------|
| **API** | JSON / XML / Protobuf responses | Fast, predictable round‑trip responses; upstream services remain responsive | **Cascading failures in synchronous chains**, timeouts, retry storms, increased latency |
| **Queue** | Individual messages | Steady consumption rate; messages processed once; queue depth remains stable | **Message duplication**, DLQ growth, consumer lag, **backpressure collapse** |
| **Pub/Sub** | Broadcast events | All subscribers receive events independently; producers remain decoupled | DLQs filling, subscriber lag, unbounded fan‑out causing load spikes |
| **Event Stream** | Ordered, append‑only events | High‑throughput sequential writes; consumers progress smoothly; ordering preserved within partitions | **Out‑of‑order delivery**, **schema drift** between producers and consumers, reprocessing side‑effects |
| **Protocols** | Encoded frames or packets | Efficient, predictable binary communication; strict framing and versioning | Packet loss, partial frames, incompatible encodings, version mismatches |
| **File‑based Integration** | Batch files (CSV, JSON, Parquet) | Simple, predictable, easy to audit; good for large batch transfers | High latency, no real‑time behaviour, operational overhead, failure discovered only after batch completes |
| **Shared Database** | Shared tables / rows | Easy integration; immediate consistency | Strong anti‑pattern: tight coupling, cross‑team schema breakage, locking/contention, unpredictable load interactions |
