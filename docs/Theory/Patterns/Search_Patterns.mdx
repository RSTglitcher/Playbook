:::tip Definition
Search Patterns describe the strategies and system behaviours used to ingest, index, retrieve, filter, and rank information efficiently across large datasets.
:::

---

## When to Use

- You need fast, relevant retrieval across large datasets
- You need faceted filtering or structured queries
- You need semantic or AI‑driven search
- You need real‑time or near‑real‑time indexing
- You need dashboards, catalogues, or log search

## When Not to Use

- You need ACID transactions
- You need relational joins
- Data is small, static, or easily scanned
- You only need key‑value lookups
- You need strict schema enforcement

---

## 🎯 What Problem Does This Solve?

Search Patterns solve the problem of **finding relevant information quickly and accurately** in large, unstructured or semi‑structured datasets.

They provide predictable ways to:

- index and analyse text
- retrieve documents efficiently
- rank results by relevance
- filter and aggregate structured fields
- perform semantic similarity search

This enables TAs to identify which search strategy fits which problem shape.

---

## 🧠 **Conceptual Model**

### **Core Components**

- **Ingestion** — documents, logs, events enter the system
- **Analysis** — tokenisation, normalisation, stemming, embeddings
- **Indexing** — inverted indexes, columnar stores, vector indexes
- **Retrieval** — keyword, boolean, filters, ranges, vector similarity
- **Ranking** — BM25, TF‑IDF, hybrid scoring, semantic reranking

### **Axes of Variation**

- Keyword vs semantic
- Exact match vs fuzzy match
- Structured vs unstructured data
- Real‑time vs batch indexing
- Precision vs recall

## 🔍 **TA Lens**

:::info How a TA Evaluates This Concept
- What changes as data grows?
- What becomes a bottleneck: indexing, retrieval, ranking?
- How does the system behave under high query load?
- How does ingestion lag affect freshness?
- Are queries keyword‑shaped, filter‑shaped, or semantic‑shaped?
- What trade‑offs exist between precision, recall, and latency?
:::

**What happens when:**

- **Data grows** → indexing slows, shards increase, relevance shifts
- **Traffic increases** → query latency rises, caches become critical
- **Concurrency rises** → hotspots appear, ranking becomes expensive
- **Resources become constrained** → ANN search degrades, merges stall

## 📘 **Key Terminology**

| Term | Definition |
|------|------------|
| **Document** | Structured object containing fields |
| **Field** | Attribute inside a document |
| **Analyzer** | Text processing pipeline |
| **Inverted Index** | Maps terms → documents |
| **Embedding** | Vector representation of meaning |
| **ANN Search** | Approximate nearest neighbour search |
| **Ranking** | Ordering results by relevance |
| **Vector Index** | Index storing embeddings for similarity search |
| **Partitioning** | Splitting data into shards or segments |

## 🧬 **Variants / Types**

<Tabs>

<TabItem value="fulltext" label="Full‑Text Search">

### Full‑Text Search

**Purpose**
Retrieve documents based on text relevance.

**Key Characteristics**
- Tokenisation
- Stemming
- Relevance scoring
- Fuzzy matching

**Behaviour**
Matches documents based on processed text and relevance scoring.

**Trade-offs**
Great for meaning; weaker for structured precision.

</TabItem>

<TabItem value="filtering" label="Filtering & Faceting">

### Filtering & Faceting

**Purpose**
Retrieve documents based on structured fields.

**Key Characteristics**
- Exact matches
- Range queries
- Boolean logic
- Faceted navigation

**Behaviour**
Filters documents using structured attributes.

**Trade-offs**
High precision; no semantic understanding.

</TabItem>

<TabItem value="analytics" label="Aggregation">

### Aggregation

**Purpose**
Compute metrics and summaries over large datasets.

**Key Characteristics**
- Bucketing
- Histograms
- Counts and metrics
- Time‑series grouping

**Behaviour**
Executes analytical queries instead of retrieving documents.

**Trade-offs**
Expensive at scale; not suitable for relevance search.

</TabItem>

<TabItem value="vector" label="Vector Search">

### Vector Search

**Purpose**
Retrieve documents based on semantic similarity.

**Key Characteristics**
- Embeddings
- ANN search
- High‑dimensional vectors

**Behaviour**
Finds nearest neighbours in vector space.

**Trade-offs**
Semantic power vs ANN approximation errors.

</TabItem>

<TabItem value="hybrid" label="Hybrid Search">

### Hybrid Search

**Purpose**
Combine keyword relevance + semantic similarity.

**Key Characteristics**
- Weighted scoring
- Multi‑stage retrieval
- Reranking

**Behaviour**
Blends keyword and vector signals for richer relevance.

**Trade-offs**
More accurate but more expensive.

</TabItem>

</Tabs>

## 🧩 **System Interactions**

:::info How a TA Understands the System
- How search interacts with ingestion pipelines
- How indexing competes with query load
- How shards, replicas, and caches affect performance
- What becomes a bottleneck as data or traffic grows
:::

### **Local Systems**

- CPU for scoring
- Memory for caching
- Disk for index segments
- Network for distributed queries

### **Remote Systems**

- Distributed clusters
- Shards and replicas
- Cross‑data‑centre search

### **Questions to Ask**

- Is indexing keeping up with ingestion?
- Are queries slow due to ranking or retrieval?
- Are shards balanced?
- Is caching effective?

## 💥 **Outputs / Results**

:::note Special Considerations
Search outputs vary by query type: documents, metrics, facets, or similarity scores.
:::

### **Success Modes**

| Result Type | Description |
|-------------|-------------|
| Ranked Results | Ordered list of relevant documents |
| Aggregations | Metrics, histograms, summaries |
| Facets | Filterable categories |
| Similarity Scores | Vector‑based relevance |

### **Failure Modes**

| Failure Type | Description |
|-------------|-------------|
| Slow Queries | Ranking or retrieval bottlenecks |
| Stale Results | Indexing lag or ingestion delays |
| Irrelevant Results | Poor analysis or ranking |
| Missing Data | Index corruption or shard imbalance |
