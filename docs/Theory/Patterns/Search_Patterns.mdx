:::tip Definition
**Search Patterns** describe the recurring strategies, concepts, and problem shapes used to index, retrieve, filter, and rank information efficiently.
They explain *how search works conceptually*, independent of any specific tool or platform.
:::

Search is not just “finding text” — it is a discipline combining **information retrieval**, **indexing**, **ranking**, and **query patterns**.

---

## 🎯 What Problem Do Search Patterns Solve?

Search Patterns enable:

| Benefit | Why it matters |
|--------|----------------|
| **Fast retrieval** | Quickly find relevant information across large datasets |
| **Relevance ranking** | Order results by usefulness, not just matching |
| **Filtering & faceting** | Support catalogues, dashboards, and analytics |
| **Scalable indexing** | Handle continuous ingestion of documents or events |
| **Semantic search** | Retrieve results based on meaning, not keywords |

Search patterns help TAs recognise *which search strategy fits which problem*.

---

## 🧠 Conceptual Model (How Search Works)

Search systems follow a predictable conceptual flow:

### **1. Ingestion Pattern**
Data enters the system as documents, events, or records.

### **2. Analysis Pattern**
Text is processed through:
- tokenisation
- normalisation
- stemming
- stopword removal
- embedding generation

### **3. Indexing Pattern**
Data is stored in structures optimised for retrieval:
- inverted indexes
- columnar stores
- vector indexes

### **4. Retrieval Pattern**
Queries match documents using:
- keywords
- filters
- boolean logic
- ranges
- aggregations
- vector similarity

### **5. Ranking Pattern**
Results are ordered using:
- term frequency
- inverse document frequency
- BM25‑style scoring
- hybrid keyword + vector ranking

# 🧩 Core Concepts in Search Patterns

### **Document Pattern**
Represent data as structured objects with fields.

### **Field Pattern**
Store attributes that can be searched, filtered, or aggregated.

### **Analyzer Pattern**
Define how text is processed before indexing.

### **Inverted Index Pattern**
Map terms → documents for fast retrieval.

### **Ranking Pattern**
Score documents based on relevance.

### **Vector Index Pattern**
Store embeddings for semantic similarity.

### **Partitioning Pattern**
Split data into shards or segments for scalability.

---

# 🧬 Query Patterns

### **Keyword Query Pattern**
Match documents containing specific terms.

### **Boolean Query Pattern**
Combine conditions using AND / OR / NOT.

### **Filter Pattern**
Match exact values or ranges.

### **Aggregation Pattern**
Compute metrics or summaries.

### **Vector Similarity Pattern**
Find nearest neighbours in embedding space.

### **Hybrid Query Pattern**
Blend keyword + vector retrieval.

---

# 🧭 When to Use Search Patterns

Use search patterns when:

- You need **fast, relevant retrieval**
- You have **large volumes of documents or logs**
- You need **faceted filtering**
- You need **semantic or AI‑driven search**
- You need **real‑time indexing**

---

# 🚫 When Not to Use

Avoid search patterns when:

- You need ACID transactions
- You need relational joins
- Data is small and static
- You only need key‑value lookups
- You need strict schema enforcement
