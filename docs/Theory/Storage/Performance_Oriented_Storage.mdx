:::tip Definition
**Performance‑Oriented Storage Systems** are specialised storage layers designed to deliver **extremely fast access**, **low latency**, or **high‑efficiency retrieval** for application workloads.
They optimise for *speed*, *searchability*, and *responsiveness* rather than durability or authoritative correctness.
:::

These systems complement — but do not replace — authoritative storage such as databases.

Performance‑oriented storage includes:

- **Caches** — ultra‑fast key/value access
- **In‑memory stores** — low‑latency data structures
- **Search indices** — full‑text and relevance‑based retrieval
- **Materialised views** — precomputed results for fast reads

---

## 🎯 What Problem Does This Solve?

Performance‑oriented storage enables:

| Benefit | Why it matters |
|--------|----------------|
| Low‑latency reads | Accelerates hot paths and user‑facing operations |
| Reduced database load | Offloads repetitive or expensive queries |
| Efficient search | Supports full‑text, fuzzy, and relevance‑based queries |
| Precomputation | Avoids recomputing expensive results |
| High throughput | Handles large read volumes with minimal overhead |

These systems are essential for responsive applications and scalable architectures.

---

## 🧠 Conceptual Model (How It Works)

Performance‑oriented storage systems optimise for:

### **Speed over durability**
- Data may be ephemeral
- Durability is optional or limited

### **Read optimisation**
- Designed for fast lookups, scans, or search queries

### **Derived or denormalised data**
- Often stores *copies* or *representations* of authoritative data

### **Specialised data structures**
- Inverted indices
- Sorted sets
- Hash maps
- Precomputed aggregates

### **Separation of concerns**
- Database = correctness
- Performance storage = speed

---

## 🔬 Behaviour Under Load / Scale

- Caches absorb read spikes and reduce database pressure
- In‑memory stores scale horizontally for high throughput
- Search indices handle complex queries efficiently
- Materialised views reduce CPU load by avoiding recomputation
- Eviction, TTLs, and invalidation strategies shape behaviour

**TA insight:**
Most performance issues arise from cache misses, stale indices, or invalidation failures — not from the storage engine itself.

---

# ⭐ **Caching Patterns (Conceptual)**
*(New, expanded section)*

Caching is the most common form of performance‑oriented storage.
It is a **speed layer**, not a system of record.

Caching introduces predictable patterns and trade‑offs that a TA must recognise.

---

## 🧩 Core Caching Patterns

### **1. Cache‑Aside (Lazy Loading)**
The application reads from the cache first.
If the value is missing, it loads from the database and writes it into the cache.

**Characteristics**
- Most common pattern
- Cache populated on demand
- Staleness controlled by TTL or explicit invalidation

**TA Lens:**
Ask: *“What happens on a cache miss?”*

---

### **2. Write‑Through**
Every write goes to the database **and** the cache at the same time.

**Characteristics**
- Cache always up‑to‑date
- Slower writes
- Simple read path

**TA Lens:**
Ask: *“Do writes update the cache synchronously?”*

---

### **3. Write‑Back (Write‑Behind)**
Writes go to the cache first and are flushed to the database later.

**Characteristics**
- Fast writes
- Risk of data loss if cache fails
- Requires careful durability guarantees

**TA Lens:**
Ask: *“Where does the authoritative write land first?”*

---

## 🧩 Cache Invalidation Strategies

Invalidation is the hardest part of caching.
A TA needs to recognise the strategy being used.

### **1. TTL‑Based Expiry**
Entries expire after a fixed time.

### **2. Explicit Invalidation**
Application removes or updates entries when data changes.

### **3. Event‑Driven Invalidation**
Changes in the database trigger cache updates.

### **4. Version‑Based Invalidation**
Cache entries include version numbers or timestamps.

### **5. Stale‑While‑Revalidate**
Serve stale data while refreshing in the background.

**TA Lens:**
Ask: *“How does this system ensure cached data stays fresh?”*

---

## 🧩 Cache Hierarchies

Caching often exists at multiple layers:

- **Browser cache**
- **CDN edge cache**
- **Reverse proxy cache**
- **Application cache**
- **In‑memory store**
- **Database buffer cache**

Each layer has different:
- TTLs
- invalidation rules
- consistency expectations

**TA Lens:**
Ask: *“Which layer is responsible for this stale or inconsistent behaviour?”*

---

## 🧩 Consistency vs Freshness Trade‑offs

Caching introduces predictable anomalies:

- stale reads
- read‑your‑writes gaps
- inconsistent views across nodes
- race conditions during invalidation

A TA must understand:

> **Caching is a deliberate trade‑off: correctness → speed.**

**TA Lens:**
Ask: *“What level of staleness is acceptable for this data?”*

---

## ⚖️ Trade-offs

| Storage Type | Strengths | Weaknesses |
|--------------|-----------|------------|
| **Caches** | Extremely fast | Not durable, requires invalidation |
| **In‑memory stores** | High throughput | Memory‑bound, volatile |
| **Search indices** | Full‑text & relevance | Not authoritative, requires indexing |
| **Materialised views** | Fast reads | Staleness unless refreshed |

---

## 🧭 When to Use Performance-Oriented Storage

Use performance‑oriented storage when:

- You need **low‑latency access** to hot data
- You need to **offload repetitive queries** from databases
- You need **full‑text or relevance‑based search**
- You need **precomputed or denormalised views**
- You need **high‑throughput read workloads**

Typical examples:
- User session data
- Feature flags
- Product search
- Autocomplete
- Leaderboards
- Precomputed analytics for dashboards

---

## 🚫 When Not to Use

Avoid performance‑oriented storage when:

- You need **authoritative, durable data**
- You need **strong consistency**
- You need **complex transactional guarantees**
- You need **large‑scale analytical queries**
- You cannot tolerate **stale or eventually consistent data**

---

## 🔍 TA Lens

:::info How a TA Evaluates Performance-Oriented Storage
- What is the access pattern (read/write/scan/search)?
- What level of staleness is acceptable?
- What invalidation strategy is required?
- Is the data derived or authoritative?
- What happens on cache miss or index miss?
- How does the system behave under eviction or memory pressure?
- How is the data refreshed or rebuilt?
:::

## 📘 Key Terminology

| Term | Definition |
|------|------------|
| Cache | Fast, in‑memory key/value store |
| TTL | Time‑to‑live for cached entries |
| Eviction | Removal of entries under memory pressure |
| Inverted Index | Structure for full‑text search |
| Materialised View | Precomputed query result |
| Staleness | Divergence from authoritative data |
| Invalidation | Removing or updating stale entries |
| Cache‑Aside | Load on miss, write to cache |
| Write‑Through | Write to cache and DB together |
| Write‑Back | Write to cache first, DB later |

# 🧬 Variants / Types

<Tabs>

<TabItem value="cache" label="Caches">

### Key Characteristics
- Ultra‑fast key/value access
- In‑memory or hybrid memory/disk
- TTL‑based or eviction‑based lifecycle

### Core Concepts
- Hot data
- Cache keys
- TTLs and expiry
- Eviction policies (LRU, LFU, FIFO)
- Cache‑aside / write‑through / write‑back

### Behaviour
- Extremely low latency
- Misses fall back to authoritative storage
- Requires careful invalidation

### Exceptions (Not Universal)
- Some caches persist to disk
- Some support limited transactions

</TabItem>

<TabItem value="inmemory" label="In-Memory Stores">

### Key Characteristics
- Data stored entirely in memory
- High throughput and low latency
- Supports complex data structures

### Core Concepts
- Hash maps
- Sorted sets
- Counters
- Pub/sub primitives

### Behaviour
- Volatile unless persisted
- Ideal for real‑time workloads

### Exceptions (Not Universal)
- Some in‑memory stores support snapshotting
- Some provide append‑only durability

</TabItem>

<TabItem value="search" label="Search Indices">

### Key Characteristics
- Full‑text search
- Relevance scoring
- Faceting and filtering

### Core Concepts
- Inverted indices
- Tokenisation
- Ranking and scoring
- Index refresh cycles

### Behaviour
- Not authoritative
- Requires indexing pipelines
- Supports complex queries efficiently

### Exceptions (Not Universal)
- Some search engines support limited transactions
- Some support near‑real‑time indexing

</TabItem>

<TabItem value="materialised" label="Materialised Views">

### Key Characteristics
- Precomputed query results
- Fast reads
- Derived from authoritative data

### Core Concepts
- Refresh intervals
- Incremental updates
- Denormalisation

### Behaviour
- May become stale
- Reduces CPU load on primary storage

### Exceptions (Not Universal)
- Some systems support automatic refresh
- Some require manual rebuilds

</TabItem>

## 🧩 System Interactions

Performance‑oriented storage interacts with:

- **Databases** → authoritative source of truth
- **Application logic** → cache reads/writes, search queries
- **Indexing pipelines** → building and refreshing search indices
- **Orchestration** → scaling, memory limits
- **Observability** → hit/miss ratios, latency, eviction metrics

**TA insight:**
Understanding hit/miss ratios, staleness windows, and invalidation paths is essential for diagnosing performance issues.

---

## 💥 Failure Modes

- Cache stampedes
- Stale or inconsistent data
- Eviction under memory pressure
- Index drift or incomplete indexing
- Hot key concentration
- Materialised view staleness
- Rebuild storms after restart
