:::tip Definition
**Hacking & Cyber Security Patterns** describe the recurring ways attackers exploit systems, and the corresponding defensive strategies used to prevent, detect, and contain those attacks.
They focus on *attack shapes*, *risk patterns*, and *protection principles*, not tools or implementation details.
:::

Cyber security is fundamentally about understanding **how systems fail under adversarial pressure** — and how to design them so they fail safely.

---

## 🎯 What Problem Does This Solve?

Understanding hacking patterns enables:

| Benefit | Why it matters |
|--------|----------------|
| **Threat awareness** | Recognise how systems are commonly attacked |
| **Risk anticipation** | Spot vulnerabilities early in design reviews |
| **Stronger requirements** | Ask the right questions before implementation |
| **Better collaboration** | Communicate clearly with security engineers |
| **Incident readiness** | Understand how attacks unfold and how to respond |

---

## 🧠 Conceptual Model (How Attacks Work)

Most cyber attacks follow a predictable lifecycle:

### **1. Reconnaissance**
Gather information about the target.

### **2. Initial Access**
Exploit a weakness to gain entry.

### **3. Privilege Escalation**
Gain higher permissions.

### **4. Lateral Movement**
Move across systems or services.

### **5. Impact**
Exfiltrate data, disrupt systems, or take control.

### **6. Persistence**
Maintain long‑term access.

Defensive patterns map to each stage.

# 🧬 Core Attack Patterns & Protection Strategies

<Tabs>

<TabItem value="injection" label="Injection Attacks">

### Attack Pattern
Untrusted input becomes executable code.

### Examples
- SQL injection
- Command injection
- Template injection
- LDAP injection

### How It Works
Attackers craft input that changes the meaning of a query or command.

### Protection Patterns
- Strong input validation
- Parameterised queries
- Escaping and output encoding
- Least privilege on data stores

### TA Lens
Ask: *“Where does untrusted input flow into interpreters?”*

</TabItem>

<TabItem value="xss" label="Cross‑Site Scripting (XSS)">

### Attack Pattern
Untrusted data is rendered as executable script in a browser.

### How It Works
Attackers inject JavaScript into pages viewed by other users.

### Protection Patterns
- Output encoding
- Content Security Policy (CSP)
- Avoid inline scripts
- Sanitise user‑generated content

### TA Lens
Ask: *“Where is untrusted data rendered into HTML?”*

</TabItem>

<TabItem value="auth" label="Authentication & Session Attacks">

### Attack Pattern
Attackers bypass or break authentication.

### Examples
- Credential stuffing
- Brute force
- Session hijacking
- Weak password resets

### Protection Patterns
- Strong authentication
- MFA
- Rate limiting
- Secure session tokens
- Safe password reset flows

### TA Lens
Ask: *“How do we prove identity, and how can that be abused?”*

</TabItem>

<TabItem value="access" label="Broken Access Control">

### Attack Pattern
Attackers access data or actions they shouldn’t.

### Examples
- IDOR (Insecure Direct Object Reference)
- Privilege escalation
- Missing authorisation checks

### Protection Patterns
- Enforce authorisation on every request
- Use server‑side checks
- Avoid trusting client‑side state

### TA Lens
Ask: *“Who should be allowed to do this, and how is that enforced?”*

</TabItem>

<TabItem value="csrf" label="Cross‑Site Request Forgery (CSRF)">

### Attack Pattern
Attackers trick a user’s browser into performing unwanted actions.

### Protection Patterns
- CSRF tokens
- SameSite cookies
- Double‑submit cookies
- Re‑authentication for sensitive actions

### TA Lens
Ask: *“Can another site trigger actions on behalf of a user?”*

</TabItem>

<TabItem value="deserialisation" label="Deserialization Attacks">

### Attack Pattern
Untrusted data is turned into objects or code.

### Protection Patterns
- Avoid unsafe deserialisation
- Validate types
- Use safe formats (JSON over binary)
- Sandbox risky operations

### TA Lens
Ask: *“Where do we turn input into objects, and is that safe?”*

</TabItem>

<TabItem value="supply" label="Supply Chain Attacks">

### Attack Pattern
Attackers compromise dependencies, packages, or build pipelines.

### Protection Patterns
- Dependency scanning
- Version pinning
- Trusted registries
- Build pipeline hardening

### TA Lens
Ask: *“What external code do we rely on, and how do we trust it?”*

</TabItem>

<TabItem value="ddos" label="Denial of Service (DoS/DDoS)">

### Attack Pattern
Attackers overwhelm systems with traffic or expensive operations.

### Protection Patterns
- Rate limiting
- Caching
- Circuit breakers
- Autoscaling
- WAF rules

### TA Lens
Ask: *“What operations are expensive, and can they be abused?”*

</TabItem>

<TabItem value="phishing" label="Phishing & Social Engineering">

### Attack Pattern
Attackers target humans, not systems.

### Protection Patterns
- User education
- MFA
- Email filtering
- Safe password reset flows

### TA Lens
Ask: *“Where do humans introduce risk?”*

</TabItem>

</Tabs>

# 🧩 Additional Attack Patterns (TA‑Friendly)

### **Misconfiguration Pattern**
Systems left in unsafe default states.

### **Sensitive Data Exposure Pattern**
Secrets, tokens, or personal data leaked.

### **Directory Traversal Pattern**
Attackers access files outside intended paths.

### **API Abuse Pattern**
Attackers exploit overly permissive or undocumented API behaviour.

### **Business Logic Abuse Pattern**
Attackers exploit *intended* behaviour in unintended ways.

---

# 🛡️ Defensive Patterns (High‑Level)

### **Least Privilege**
Give only the permissions required.

### **Zero Trust**
Never trust by default — always verify.

### **Defense in Depth**
Multiple layers of protection.

### **Secure Defaults**
Systems should be safe without configuration.

### **Fail Closed**
If something breaks, access should be denied.

### **Threat Modelling**
Identify what attackers want and how they might get it.

---

# 🧭 When to Use These Patterns

Use hacking & security patterns when:

- Reviewing designs
- Evaluating APIs
- Assessing data flows
- Working with authentication or authorisation
- Reviewing third‑party integrations
- Handling sensitive data
- Preparing for audits or compliance

---

# 🚫 When Not to Use

Avoid over‑security when:

- The system is isolated and non‑critical
- The cost of controls outweighs the risk
- You are still prototyping and not storing real data

Security should be **proportionate**, not maximal.

# 📘 Key Terminology

| Term | Definition |
|------|------------|
| **Attack Surface** | All points where an attacker can interact |
| **Threat Model** | Assumptions about attacker capabilities |
| **Payload** | Malicious input crafted to exploit a vulnerability |
| **Vector** | The path an attack takes |
| **Exploit** | Code or technique that triggers a vulnerability |
| **Mitigation** | A control that reduces risk |

# 🏗️ Real‑World Examples

- A login endpoint rate‑limited to prevent brute force
- A product API enforcing authorisation on every request
- A form rejecting unsafe characters
- A system encrypting tokens before storage
- A pipeline scanning dependencies for vulnerabilities
- A WAF blocking suspicious traffic patterns
