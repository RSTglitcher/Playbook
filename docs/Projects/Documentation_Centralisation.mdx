:::tip 📌 Executive Summary

**MM-YYYY:** Jan-2023 to Dec-2025

**Project Purpose**  
To centralise, automate, and govern department documentation by creating a single, authoritative documentation experience
that supports self‑service discovery, onboarding, issue investigation, and eventually is a source of truth for AI training.

**Problem Statement**  
Documentation is fragmented, manually maintained, and inconsistently owned, causing it to become outdated quickly,
difficult to trust, and expensive to maintain, which slows investigations, onboarding, and effective self‑service.

**Value Added:**
- Researched available open-source documentation websites
- Set up a basic site to scrape GitHub Documentation and wrote onboarding pages for more specific requests
- Centralised Access & Permissions documetation to faciliate onboarding
- Added a homepage with links to all department specific platforms to encourage a single point of discovery that is persona agnostic
:::

## 🧭 **Context & Background**

Capture the “why now?” and inherited constraints.

- **Trigger or catalyst:**  
  Increasing reliance on self‑service tooling (DataHub, platform services) and repeated user support questions highlighted gaps in documentation discoverability and freshness.

- **Relevant history:**  
  - 2023: Creation of the `gottdata` documentation site via automated scraping
  > - Addition of a User Guide layer for accessibility
  > - Identification of local build/launch as a usability barrier
  - 2024: Decision to host documentation under Global Data Catalogue User Guides
  > - Automate content refreshes with successful master branch rebuilds

- **Domain boundaries involved:**  
  Documentation, metadata, developer enablement, user support

- **Assumptions already in place:**  
  Documentation should be treated as a product, not a by‑product of delivery

- **Known constraints:**  
  Legacy documentation quality varies widely across teams

## 🔍 **Analysis Plan**

### **Key Questions to Resolve**

- What documentation should be considered authoritative?
- Where should documentation be authored vs discovered?
- Who owns documentation accuracy and freshness?
- How can updates be automated safely?

### **Investigation Steps**

- Audit existing documentation sources  
- Identify duplication, gaps, and stale content  
- Define content hierarchy and taxonomy  
- Design opt‑in / opt‑out mechanisms  
- Validate governance approach with stakeholders

### **Artefacts Produced**

- Documentation taxonomy  
- Automation scripts  
- Site structure and navigation  
- Ownership and workflow definitions  
- Decision records

## 🧩 **Architecture & Design Notes**

Capture reasoning, not just conclusions.

- **Communication pattern choices:**  
  Pull‑based automation from source repositories

- **Storage decisions:**  
  GitHub retained as source of truth; presentation layers derived

- **Integration boundaries:**  
  Source documentation vs catalogue discovery

- **Failure modes:**  
  - Stale documentation  
  - Unowned content  
  - Fragmented publishing paths  

- **Trade‑offs considered:**  
  Manual curation vs automation; flexibility vs consistency

- **Alternatives rejected (and why):**  
  Fully manual maintenance due to sustainability risk
