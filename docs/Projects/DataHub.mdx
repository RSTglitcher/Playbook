:::tip 📌 Executive Summary
**MM-YYYY:** Oct-2023 to Dec-2025

**Project Purpose**
- MVP: To design, plan, and deliver a Global Data Catalogue (GDC) capability using DataHub as the underlying platform, enabling
consistent discovery, governance, lineage, and ownership of data assets across the department ecosystem.
- Iterations: To evolve **DataHub** as the central capability for discovering, governing, and understanding data across the organisation
by enabling consistent, trusted **metadata capture, publishing, and consumption** via a dedicated software engineering team.

**Problem Statement**
Metadata across platforms is inconsistent, fragmented, or implicit, making it difficult for users, systems, and governance
processes to reliably understand what data exists, what it represents, and how it should be used and preventing the department
from operating as a truly self‑service data platform.

**Value Added:**

## 🧭 **Context & Background**
This work spans **nearly two years** and represents a foundational capability for the department rather than a point solution.
- **Trigger / Catalyst:**
  Growth in data volume, producers, and consumers exposed the absence of a single, trusted way to understand *what data exists*, *where it comes from*, and *how it should be used*.
- **Relevant History:**
  - Early reliance on spreadsheets, wiki pages, and bespoke documentation
  - Adoption of DataHub as an open‑source metadata platform
  - Evolution toward a branded **Global Data Catalogue (GDC)** experience aligned to department needs
  - Progressive delivery across multiple roadmap phases (2024–2026)
- **Domain Boundaries Involved:**
  Platform engineering, data producers, analytics consumers, governance and compliance
- **Assumptions Already in Place:**
  - Metadata is a pre‑requisite for trust and self‑service
  - Producers are accountable for what they publish; consumers for how they use it

- **Known Constraints:**
  - Partial metadata availability from legacy systems
  - Varied maturity across producer teams

## 🎯 **Scope**

Make metadata boundaries explicit.

- **In Scope:**
    - Metadata discovery, search, and lineage
    - Dataset ownership, categorisation, and documentation
    - Feed registration and self‑service workflows
    - Planning and phased rollout of GDC capabilities
    - Integration patterns for metadata events

- **Out of Scope:**
  - Physical data transformation logic
  - BI-layer metric definitions

- **Assumptions:**
  - Metadata can be treated as a product
  - Consumers rely on GDC as the source of truth
  - Metadata quality improves through enablement, not enforcement alone.

- **Constraints:**
  - Partial metadata availability from legacy sources
  - Existing producer workflows and operational overhead

- **Non‑Goals:**
  - Replacing source systems
  - Solving all data quality issues

## 🔍 **Analysis Plan**

### **Key Questions to Resolve**

- What metadata must exist for DataHub consumers to trust the data?
- Where should metadata be authored vs inferred?
- How does metadata move between team platforms/tools and GDC?
- What failure modes break discoverability or trust?
- How do we enable self‑service without overwhelming producers?
- Which capabilities must exist before governance can scale?
- How should GDC evolve across roadmap phases?

### **Investigation Steps**

- Review existing metadata schemas and usage
- Identify gaps, duplication, and ambiguity
- Review existing metadata and documentation gaps
- Define core entities, ownership models, and lineage expectations
- Shape phased delivery plans aligned to roadmap constraints
- Validate with stakeholders through demos and showcases

## 🧩 **Architecture & Design Notes**

- **Communication Pattern Choices:**
  Progressive enablement via UI, APIs, and documentation

- **Integration Boundaries:**
  Clear separation between source systems and metadata platform

- **Failure Modes Considered:**
  - Catalogue as documentation graveyard
  - Incomplete lineage reducing trust
  - Producer disengagement

- **Trade‑offs Considered:**
  Speed of onboarding vs consistency of metadata

- **Alternatives Rejected (and why):**
  Bespoke, team‑specific catalogues that do not scale organisationally
