## 🛠️ Workbench
This **Workbench** space contains technical mediums analysts use to think, reason, design, and build.
It brings together programming languages, AI models, frameworks, and emerging tools into a single, coherent space.

Where **Theory** explains how systems work, **Runbook** explains how tools behave, and **Projects** show how these were applied —
**Workbench captures *how you think and work*.**

It is the living catalogue of your cognitive environments:
- **Theory** → conceptual understanding
- **Runbook** → operational behaviour
- **Projects** → applied execution

It strengthens your ability to:
- choose the right medium for the problem
- interpret engineering decisions
- validate feasibility
- communicate effectively with humans and AI
- design systems with clarity and precision

| Step No. | Step | Thought Process (Language‑Agnostic) | Example A: Simple Domain | Example B: Backend / Platform |
|---------|------|--------------------------------------|---------------------------|-------------------------------|
| **1** | **Start with the Problem Statement** | Identify the *intent*, *actors*, and *outcomes*. Engineers begin with the same artefact analysts produce. | “A customer can place an order containing multiple products.” | “When a product is updated, publish an event to Kafka so downstream systems can update their state.” |
| **2** | **Extract Domain Concepts (Nouns → Entities)** | Nouns become **classes**, **components**, or **services**. | Customer, Order, Product | ProductUpdatedEvent, CatalogService, KafkaProducer, SearchIndexUpdater |
| **3** | **Extract Behaviours (Verbs → Methods / Responsibilities)** | Verbs become **methods**, **handlers**, or **operations**. | placeOrder(), calculateTotal() | publishEvent(), handleEvent(), updateIndex(), recalculateRecommendations() |
| **4** | **Identify Required Data (Attributes → Fields / Schemas)** | Determine what each entity needs to store to perform its behaviour. | Product → name, price; Order → List`<Product`> | Event → productId, timestamp, changedFields; Cache → productSnapshot |
| **5** | **Choose Structure (Class / Interface / Inheritance / Component)** | Decide how to model the domain: identity, capability, or specialisation. | Product → class; Discountable → interface | Event → immutable class; Producer/Consumer → interfaces; Handlers → concrete classes |
| **6** | **Choose Data Structures (List / Set / Map / Queue / Stream)** | Choose based on behaviour: ordering, uniqueness, lookup, streaming. | Order.products → List`<Product`> | Kafka → distributed queue; Cache → Map`<Id, Snapshot>`; Search → inverted index |
| **7** | **Define Object Lifecycle (Constructors, DI, Factories)** | Ensure objects/components start in a valid state. | new Product(name, price) | new ProductUpdatedEvent(productId, timestamp); DI for KafkaProducer |
| **8** | **Build the First Working Slice (Walking Skeleton)** | Build the smallest end‑to‑end flow that works. | Create order → add product → calculate total | CatalogService publishes event → KafkaProducer sends → one consumer handles |
| **9** | **Test as You Build (Unit → Integration → Contract → E2E)** | Validate behaviour incrementally. Tests guide design. | JUnit test for calculateTotal() | Unit tests for handlers; Contract tests for schemas; Integration tests for Kafka; E2E for search index update |
| **10** | **Iterate: Build → Run → Observe → Adjust** | Engineers plan one small step, run it, adjust. This *is* the plan. | Add discounts → test → refine | Add retries, DLQs, idempotency, caching → test each layer |
| **11** | **Expand the Model Safely (Features, Edge Cases, Errors)** | Add complexity only after the core works. | Add validation, error handling | Add schema evolution, observability, concurrency controls |
| **12** | **Integrate with the Wider System (Architecture Thinking)** | Consider interactions, failure modes, performance, contracts. | Order system integrates with payment, inventory | Event flows integrate with search, recommendations, analytics, caches, APIs |

</details>

<details>
<summary>AI</summary>

AI in the Workbench splits into **two complementary groups**:

### **Assistants**
These are **cognitive partners** you work with. **Thinking environments** — tools that amplify reasoning, writing, coding, and analysis.


- Microsoft Copilot
- Claude
- ChatGPT
- GitHub Copilot
- Augment


### **Skills**
These are composable **intelligence infrastructure** used inside AI systems. These behave like **frameworks** as you combine them to build pipelines, retrieval systems, and intelligent features.

- Embeddings
- Vector Stores
- Tokenisation
- Model Serving
- Evaluation


| Step No. | Step | Thought Process (Language‑Agnostic) | Example A: Simple Domain | Example B: Backend / Platform |
|---------|------|--------------------------------------|---------------------------|-------------------------------|
| **1** | **Start with the Intent** | Identify the *goal*, *audience*, and *format*. A prompt is a specification. | “Explain how an order total is calculated.” | “Generate a Kafka event schema for ProductUpdatedEvent.” |
| **2** | **Extract the Core Entities** | Identify the nouns: what the prompt is *about*. | Order, Product, Discount | Event, Producer, Consumer, Schema, Topic |
| **3** | **Define the Required Output Shape** | Decide the structure: list, table, code, explanation, diagram. | “Provide a step‑by‑step explanation.” | “Return JSON schema + example payload + validation rules.” |
| **4** | **Specify Constraints** | Add boundaries: length, tone, format, exclusions, assumptions. | “Keep it under 200 words.” | “Use Avro schema; include backward‑compatibility notes.” |
| **5** | **Provide Context** | Give the model the information it needs to reason correctly. | “Assume products have price and quantity.” | “Assume Kafka topic is partitioned by productId.” |
| **6** | **Define the Role or Perspective** | Tell the model *how* to think: TA, engineer, architect, PM. | “Explain as if teaching a junior developer.” | “Explain as a senior backend engineer designing event contracts.” |
| **7** | **Add Examples (Few‑Shot Prompting)** | Show the model what “good” looks like. | Provide a sample order → expected total. | Provide a sample event → expected consumer behaviour. |
| **8** | **Ask for the Output** | Make the request explicit and unambiguous. | “Now generate the explanation.” | “Now generate the schema + consumer contract.” |
| **9** | **Review the Output** | Check for correctness, completeness, and alignment with intent. | Does the explanation match the domain? | Does the schema match platform constraints? |
| **10** | **Refine with Iterative Prompts** | Prompting is iterative: adjust, clarify, constrain. | “Rewrite with simpler language.” | “Add idempotency rules + retry semantics.” |
| **11** | **Add Edge Cases** | Ensure the model handles non‑happy paths. | “What if the product list is empty?” | “What if the event is replayed or arrives out of order?” |
| **12** | **Integrate with the Wider System** | Connect the prompt output to documentation, code, or architecture. | Add explanation to onboarding docs. | Add schema to OpenAPI, DataHub, or event catalog. |


</details>
