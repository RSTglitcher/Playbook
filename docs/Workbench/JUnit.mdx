::tip JUnit
A Java unit testing framework used to express expected behaviour, validate logic, and drive design through tests.

- **Year developed:** 1997
- **Primary purpose:** Provide a structured environment for writing automated unit tests
- **Target problem:** Validate small, isolated units of logic and support TDD workflows
:::

---

## 🗺️ Context

### History
- Created by Kent Beck and Erich Gamma (founders of XP and the Gang of Four)
- One of the original xUnit family frameworks
- Became the de‑facto standard for Java testing
- JUnit 5 introduced modular architecture, modern annotations, and extensibility
- Used across enterprise Java, Spring, microservices, and backend systems

### Strengths & Limitations

**Strengths**
- Fast, deterministic feedback
- Encourages modular, testable design
- Clear, expressive assertions
- Integrates with Maven, Gradle, IDEs, CI/CD
- Supports TDD (Red → Green → Refactor)

**Limitations**
- Only tests in‑process Java logic
- Not suitable for UI, network, or distributed tests
- Requires discipline to avoid brittle tests
- Can be misused for integration tests

### TA‑Relevant Insights
JUnit helps analysts understand:

- how developers express expected behaviour
- how edge cases and invariants are captured
- how regressions are prevented
- how method contracts are shaped
- how failures reveal system assumptions
- how testability influences architecture

JUnit is a **thinking environment**, not just a testing tool.

## 🧠 Mental Model

### How to think in this environment
JUnit expresses behaviour through a simple structure:

- **Arrange** — set up the state
- **Act** — execute the logic
- **Assert** — verify the outcome

In TDD, this becomes:

1. Write a failing test (Red)
2. Write minimal code to pass it (Green)
3. Refactor safely (Refactor)

JUnit is where developers:

- define expected behaviour
- shape method signatures
- clarify edge cases
- document intent
- enforce correctness

### Ecosystem
JUnit interacts with:

- **Mockito** (mocking dependencies)
- **AssertJ / Hamcrest** (fluent assertions)
- **Spring Test** (context testing)
- **Maven / Gradle** (build + test lifecycle)
- **JaCoCo** (coverage)
- **CI/CD pipelines** (GitHub Actions, Jenkins, Azure DevOps)

## 🔁 Common Patterns

### Idioms and Typical Approaches
- One behaviour per test
- Descriptive test names (behaviour‑focused)
- Use mocks to isolate dependencies
- Parameterised tests for variants
- Keep tests deterministic and fast
- Use expressive assertions

### Pitfalls
- Over‑mocking (brittle tests)
- Testing implementation instead of behaviour
- Large test classes with mixed responsibilities
- Slow tests due to unnecessary I/O
- Missing edge‑case coverage

## 🧪 Examples

### Behaviour‑Focused Test

```java
@Test
void returnsEmptyListWhenNoUsersFound() {
    UserRepository repo = new InMemoryUserRepository();
    assertTrue(repo.findAll().isEmpty());
}
```

### Parameterised Test

```java
@ParameterizedTest
@ValueSource(strings = {"", " ", "\n"})
void rejectsBlankUsernames(String input) {
    assertThrows(InvalidUsernameException.class, () -> User.create(input));
}
```

### Using Mockito

```java
@Test
void sendsEmailOnRegistration() {
    EmailService email = mock(EmailService.class);
    Registration reg = new Registration(email);

    reg.register("rebecca@example.com");

    verify(email).sendWelcome("rebecca@example.com");
}
```

## 📝 Glossary

| Term | Definition |
|------|------------|
| **Unit Test** | A test for a small, isolated piece of logic. |
| **Assertion** | A statement that must be true for the test to pass. |
| **Mock** | A fake dependency used to isolate behaviour. |
| **Fixture** | Setup data or objects used in tests. |
| **Red/Green/Refactor** | The TDD cycle. |
| **Parameterized Test** | A test that runs with multiple inputs. |
