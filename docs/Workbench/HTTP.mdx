:::tip HTTP
HTTP (HyperText Transfer Protocol) is the foundational communication protocol of the web.  
It defines *how* clients and servers exchange requests and responses.

- **Year developed:** 1991  
- **Primary purpose:** ClientвЂ“server communication  
- **Target problem:** Standardising how resources are requested, transferred, and represented  
:::

## рџ—єпёЏ Context

### History
HTTP was created by **Tim BernersвЂ‘Lee** alongside HTML as part of the original web architecture.  
It evolved through:

- **HTTP/0.9** вЂ” simple GET requests  
- **HTTP/1.1** вЂ” persistent connections, headers, caching  
- **HTTP/2** вЂ” multiplexing, binary framing  
- **HTTP/3** вЂ” QUIC transport, faster and more reliable  

|Code   |Status Meaning |
|------|-------------------------------------------------------------|
| **1xx** | Informational вЂ” Continue / Switching Protocols / Processing  |
| **2xx** | Success вЂ” OK / Created / Accepted / No Content / MultiвЂ‘Status / etc. |
| **3xx** | Redirection вЂ” Multiple Choices / Moved Permanently / Found / See Other / etc. |
| **4xx** | Client Error вЂ” Bad Request / Unauthorized / Forbidden / Not Found / etc. |
| **5xx** | Server Error вЂ” Internal Server Error / Not Implemented / Bad Gateway / Service Unavailable / etc. |

HTTP underpins:

- Web browsers  
- APIs  
- Microservices  
- Mobile apps  
- IoT devices  
- Cloud services  

### Strengths & Limitations

**Strengths**
- Simple, universal, wellвЂ‘understood  
- Stateless (scales horizontally)  
- Flexible payloads (JSON, HTML, binary)  
- Extensible via headers  
- Works across all platforms  

**Limitations**
- Statelessness requires additional mechanisms for sessions  
- Latency can be high in HTTP/1.1  
- Verb semantics are often misused  
- Caching behaviour can be complex  

### TAвЂ‘Relevant Insights
- HTTP reveals **how systems communicate**  
- Understanding HTTP helps you debug APIs, microservices, and integrations  
- Many issues surface in headers, status codes, or payloads  
- HTTP is the backbone of REST, GraphQL, and most modern service architectures

## рџ§  Mental Model

### **How to think in this environment**
- HTTP is **request в†’ response**  
- Clients initiate; servers respond  
- Everything is a **resource**  
- Methods express **intent** (GET, POST, PUT, DELETE)  
- Headers carry **metadata**  
- Status codes express **outcome**  
- Bodies carry **data**  

### Ecosystem
- **Methods:** GET, POST, PUT, PATCH, DELETE  
- **Status codes:** 2xx, 3xx, 4xx, 5xx  
- **Headers:** ContentвЂ‘Type, Authorization, Accept  
- **Payloads:** JSON, XML, HTML, binary  
- **Auth:** OAuth, JWT, API keys  
- **Transport:** TCP (HTTP/1.1, HTTP/2), QUIC (HTTP/3)

## рџ”Ѓ Common Patterns

### Idioms
- RESTful resource naming
- JSON as the default payload
- Stateless interactions
- Pagination via query params
- Versioning via headers or URL
- Caching with `ETag` and `Cache-Control`

### **вљ пёЏ Pitfalls**
- Misusing POST for everything
- Returning 200 OK for errors
- Missing or incorrect `Content-Type`
- Overloaded endpoints with unclear semantics
- Leaking sensitive data in responses
- Ignoring idempotency rules
