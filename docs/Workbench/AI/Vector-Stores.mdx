:::tip Vector Stores
**What they are:**
Databases optimised for storing embeddings and performing similarity search.

- **Year developed:** 2018–present
- **Primary purpose:** Fast retrieval of semantically similar items
- **Target problem:** “How can we search by meaning instead of keywords?”
:::

---

## 🗺️ Context

### History
Vector stores emerged as embeddings became central to AI systems:

- Early ANN (Approximate Nearest Neighbour) libraries → FAISS, Annoy
- First vector databases → Pinecone, Milvus
- Modern integrated stores → Weaviate, Chroma, Elasticsearch vector search

They now power:

- RAG
- semantic search
- recommendation systems
- anomaly detection
- clustering
- personalisation
- knowledge retrieval

### Strengths
- Extremely fast similarity search
- Scales to millions or billions of vectors
- Supports metadata filtering
- Integrates with RAG pipelines
- Enables semantic search and ranking

### Limitations
- Requires embeddings of consistent dimension
- Sensitive to embedding quality
- Needs careful indexing and tuning
- Harder to debug than keyword search
- Retrieval quality depends on vector density and distribution

### TA‑Relevant Insights
Vector stores matter for Technical Analysts because they:

- determine which documents an AI system retrieves
- explain why certain results appear (or don’t)
- influence the quality of RAG responses
- reveal how personalisation systems match users to content
- expose issues like drift, stale embeddings, or poor indexing
- help diagnose relevance, ranking, and retrieval failures

Vector stores are the **semantic backbone** of AI systems.

## 🧠 Mental Model

### How to Think in Vector Stores
Vector stores are not relational databases.
They are **geometric search engines**.

Think of a vector store as:

- a map of meaning
- where each item is a point in high‑dimensional space
- and search is “find the closest points to this meaning”

Similarity is measured using:

- cosine similarity
- dot product
- Euclidean distance

Vector stores let AI systems:

- retrieve relevant documents
- match users to items
- cluster similar content
- detect anomalies
- power semantic search

### Ecosystem
Vector stores interact with:

- embeddings
- RAG pipelines
- LLMs
- metadata filters
- ranking algorithms
- caching layers
- ETL pipelines

They are the **retrieval layer** of AI systems.

### Engineering Workflows

```shell
--------------
Vector Store Workflows
--------------

# Insert a vector
INSERT INTO vector_store (id, vector, metadata);

# Query by similarity
SELECT *
FROM vector_store
ORDER BY vector <-> query_embedding
LIMIT 5;

# Filter by metadata
SELECT *
FROM vector_store
WHERE metadata.category = 'payments'
ORDER BY vector <-> query_embedding;

# Update embeddings
UPDATE vector_store
SET vector = new_embedding
WHERE id = 'doc_123';
```

These workflows underpin semantic search and RAG retrieval.

## 🔁 Common Patterns

### Idioms & Typical Approaches
- “Find similar items…” → semantic search
- “Retrieve context…” → RAG
- “Cluster these…” → grouping
- “Match user to content…” → personalisation
- “Rank by similarity…” → relevance scoring
- “Filter by metadata…” → hybrid search

### Pitfalls
- Mixing embeddings from different models
- Using vector search without metadata filters
- Forgetting to re‑embed after data changes
- Assuming vector search is deterministic
- Using vector stores for exact matching
- Ignoring embedding drift over time
