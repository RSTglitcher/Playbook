:::tip Model Serving
**What it is:**
The infrastructure and processes that expose a trained model as an API or service.

- **Year developed:** 2015–present
- **Primary purpose:** Run models reliably at scale
- **Target problem:** “How do we turn a trained model into a production system?”
:::

---

## 🗺️ Context

### History
Model serving evolved from early ML scripts into full production systems:

- Early ML → cron jobs, batch scoring
- Microservices era → model APIs
- Cloud ML → managed endpoints (SageMaker, Vertex AI)
- Modern AI → GPU‑accelerated, multi‑model, low‑latency serving

Today, model serving powers:

- recommendation engines
- fraud detection
- personalisation
- ranking and scoring
- real‑time predictions
- RAG pipelines
- LLM inference

### Strengths
- Enables real‑time predictions
- Scales with traffic
- Supports versioning and rollback
- Integrates with monitoring and observability
- Can run on CPU, GPU, or specialised hardware

### Limitations
- Requires careful optimisation
- Sensitive to latency and load
- Needs robust monitoring
- Can be expensive at scale
- Requires coordination with data pipelines

### TA‑Relevant Insights
Model serving matters for Technical Analysts because it:

- determines how predictions reach downstream systems
- explains latency, throughput, and reliability constraints
- reveals how models interact with APIs, queues, and pipelines
- exposes failure modes like drift, stale features, or cold starts
- helps evaluate feasibility of ML‑powered features
- informs design decisions around sync vs async predictions

Model serving is the **runtime reality** of ML systems.

## 🧠 Mental Model

### How to Think in Model Serving
Model serving is not training — it is **systems engineering**.

Think of model serving as:

- a microservice
- that loads a model
- receives requests
- runs inference
- returns predictions
- under strict latency and reliability constraints

It is where:

- ML meets distributed systems
- predictions meet SLAs
- data freshness meets performance
- scaling meets cost

### Ecosystem
Model serving interacts with:

- feature stores
- data pipelines
- APIs
- queues
- GPUs/CPUs
- monitoring systems
- autoscaling
- caching layers
- model registries

It is the **operational backbone** of ML.

## 🔁 Common Patterns

### Idioms & Typical Approaches
- “Serve as an API…” → real‑time inference
- “Batch scoring…” → offline predictions
- “Shadow deployment…” → safe rollout
- “A/B testing…” → model comparison
- “Autoscaling…” → traffic handling
- “Caching…” → reduce latency
- “GPU serving…” → heavy models

### Pitfalls
- Cold starts causing latency spikes
- Serving stale models or features
- Missing monitoring → silent failures
- Overloading GPUs/CPUs
- Using sync serving where async is needed
- Not versioning models properly
- Ignoring drift and degradation

## 📝 Glossary

| Term | Meaning |
|------|---------|
| **Inference** | Running a model to generate predictions |
| **Endpoint** | API exposing the model |
| **Cold Start** | Delay when a model loads for the first time |
| **Autoscaling** | Automatically adjusting compute based on load |
| **Shadow Deployment** | Testing a model without affecting users |
| **Model Registry** | Version control for models |
| **Feature Store** | System for managing model inputs |
