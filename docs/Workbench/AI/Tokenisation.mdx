:::tip Tokenisation
**What it is:**
The process of converting text into tokens — the atomic units models operate on.

- **Year developed:** 2012–present
- **Primary purpose:** Convert text into a numerical form models can process
- **Target problem:** “How can a model understand and compute over human language?”
:::

---

## 🗺️ Context

### History
Tokenisation evolved alongside NLP:

- Early NLP → whitespace tokenisation
- Word2Vec era → word‑level tokens
- BPE (Byte Pair Encoding) → subword tokens
- SentencePiece → language‑agnostic tokenisation
- Modern LLMs → byte‑level and multimodal tokenisation

Today, tokenisation is central to:

- LLM inference
- RAG
- embeddings
- cost estimation
- context window management
- multilingual processing

### Strengths
- Efficient representation of text
- Handles rare or unknown words
- Works across languages
- Enables compression of long text
- Predictable cost and context usage

### Limitations
- Not human‑interpretable
- Different models use different tokenisers
- Token boundaries can be surprising
- Tokenisation errors can break meaning
- Impacts cost and latency significantly

### TA‑Relevant Insights
Tokenisation matters for Technical Analysts because it:

- determines how much text fits in a context window
- affects cost and performance of AI features
- influences retrieval quality in RAG
- explains why some prompts behave unexpectedly
- reveals why certain languages cost more to process
- helps diagnose truncation, overflow, and hallucination risks

Tokenisation is the **computational substrate** of LLMs.

## 🧠 Mental Model

### How to Think in Tokenisation
Tokenisation is not about words — it is about **units of meaning the model can predict**.

Think of tokenisation as:

- the “alphabet” of the model
- the smallest building blocks of its internal language
- the units it counts, prices, and reasons over

Examples:

- “authentication” → may become 3–5 tokens
- “login” → often 1 token
- emojis → often 1 token
- rare words → many tokens
- code → highly token‑dense

### Ecosystem
Tokenisation interacts with:

- embeddings
- vector stores
- RAG pipelines
- prompt engineering
- model serving
- cost estimation
- context windows

It is the **entry point** to all LLM computation.

## 🔁 Common Patterns

### Idioms & Typical Approaches
- “Count tokens…” → cost estimation
- “Compress this…” → context optimisation
- “Rewrite shorter…” → prompt efficiency
- “Detect overflow…” → debugging
- “Compare token counts…” → multilingual analysis
- “Chunk text…” → RAG preparation

### Pitfalls
- Assuming tokens = words
- Mixing tokenisers across models
- Ignoring token limits
- Overloading prompts with unnecessary text
- Forgetting that code is token‑dense
- Assuming all languages tokenise equally

## 📝 Glossary

| Term | Meaning |
|------|---------|
| **Token** | The smallest unit of text a model processes |
| **Tokeniser** | The algorithm that splits text into tokens |
| **BPE** | Byte Pair Encoding, a common tokenisation method |
| **Context Window** | Maximum number of tokens a model can handle |
| **Token Density** | How many tokens a piece of text produces |
| **Compression** | Reducing token count while preserving meaning |
