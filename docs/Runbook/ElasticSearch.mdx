:::tip Elasticsearch  
Elasticsearch is a **distributed search and analytics engine** built on top of Apache Lucene.  
It enables **full‑text search**, **log analytics**, **time‑series analysis**, and **real‑time querying** across large datasets.  
It solves problems related to **search relevance, indexing, distributed querying, and horizontal scaling**.  
:::

### Key Terms

- **Index:** Logical namespace for documents (similar to a database table)  
- **Document:** JSON object stored and indexed  
- **Shard:** A partition of an index (primary or replica)  
- **Cluster:** A group of Elasticsearch nodes  
- **Node:** A single Elasticsearch server  
- **Mapping:** Schema definition for fields  
- **Analyzer:** Text processing pipeline (tokenization, stemming, etc.)  
- **Query DSL:** JSON-based query language for Elasticsearch  

### Knowing when to use **Elasticsearch**

| When to use | When not to use |
|-------------|-----------------|
| Full‑text search, fuzzy matching, relevance scoring | Strict relational consistency (use SQL DB) |
| Log analytics, observability, time‑series search | Heavy OLTP workloads |
| Real‑time search across large datasets | Long‑term cold storage (use S3/ADLS) |
| Autocomplete, suggestions, ranking | Complex joins or transactions |
| Distributed, scalable search workloads | Small datasets where simple DB queries suffice |

## 📘 About / Foundations  
_Conceptual grounding for how the tool works._

### How It Works (Conceptual Model)

- Data is stored as **JSON documents** inside **indices**  
- Indices are split into **shards**, distributed across nodes  
- Elasticsearch uses **inverted indices** for fast text search  
- Queries are executed using the **Query DSL**  
- Clusters automatically rebalance shards and replicas  
- **Mappings** define field types and indexing behaviour  
- **Analyzers** process text into searchable tokens  

### System Contract

Elasticsearch enforces contracts through:

- **Index mappings** (field types, analyzers, keyword vs text fields)  
- **Cluster topology** (number of nodes, shard allocation)  
- **Replication strategy** (primary + replica shards)  
- **Query DSL** (structured search queries)  
- **Index lifecycle policies** (rollover, retention, deletion)  

### Key Strategies

<Tabs>
<TabItem value="indexing" label="Indexing Strategy">

- Define mappings explicitly (avoid dynamic mapping in production)  
- Use `keyword` for exact matches, `text` for full‑text search  
- Use appropriate analyzers for language‑specific search  
- Responsibility: SWE + TA validation  
- Goal: predictable search behaviour and performance  

</TabItem>

<TabItem value="sharding" label="Sharding Strategy">

- Choose shard count based on data size and query patterns  
- Avoid oversharding (too many small shards)  
- Use replicas for high availability  
- Responsibility: Platform + SWE  
- Goal: balanced cluster, predictable performance  

</TabItem>

<TabItem value="lifecycle" label="Data Lifecycle Strategy">

- Use ILM (Index Lifecycle Management) for rollover and retention  
- Move old data to warm/cold tiers  
- Delete expired indices automatically  
- Responsibility: SWE + TA  
- Goal: cost‑efficient storage and stable cluster health  

</TabItem>
</Tabs>

### What Results Normally Look Like

- Search results with relevance scores  
- Aggregations (histograms, terms, metrics)  
- Cluster health (`green`, `yellow`, `red`)  
- Shard allocation and node distribution  
- Index statistics (docs count, size, refresh rate)  
- Slow query logs

<details>
<summary>Pre‑Requisites</summary>

#### Security
- Elasticsearch credentials (basic auth, API keys)  
- TLS/SSL configuration  
- Role‑based access control (RBAC)  
- Kibana access (if applicable)

- Node IPs and cluster endpoints  
- VPC/subnet configuration

- Load balancer or coordinator nodes

- Cluster size and node roles (master, data, ingest, coordinating)  
- Shard and replica configuration  
- Index lifecycle policies  
- Snapshot/restore configuration  

#### Data
- Mappings and field types  
- Analyzer configuration  
- Expected query patterns  
- Data ingestion pipelines (Logstash, Beats, custom apps)

#### Operational Guarantees
- Replication factor  
- Refresh interval  
- Query latency expectations  
- Cluster failover behaviour  

#### Observability Metrics
- Cluster health  
- Shard allocation  
- Node CPU/memory/disk  
- Query latency  
- Indexing throughput  
- JVM heap usage

Common workflows and commands used by engineers when operating this tool

```shell
--------------
Key Terminal Commands
--------------

# Create an index with mapping
curl -X PUT https://<host>:9200/my_index -H 'Content-Type: application/json' -d '{
  "mappings": {
    "properties": {
      "message": { "type": "text" },
      "timestamp": { "type": "date" }
    }
  }

# Index a document
curl -X POST https://<host>:9200/my_index/_doc -H 'Content-Type: application/json' -d '{
  "message": "hello world",
  "timestamp": "2024-01-01T00:00:00Z"
}'

# Search documents
curl -X GET https://<host>:9200/my_index/_search -H 'Content-Type: application/json' -d '{
  "query": { "match": { "message": "hello" } }
}'

# Run an aggregation
curl -X GET https://<host>:9200/my_index/_search -H 'Content-Type: application/json' -d '{
  "size": 0,
  "aggs": { "messages": { "terms": { "field": "message.keyword" } } }
}'
```

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|--------|---------------------|---------------|-------------------|
| Cluster red | Missing primary shards | `_cluster/health`, `_cat/shards` | Restore shards, fix node failures |
| Cluster yellow | Missing replica shards | Replica count, node availability | Add nodes, fix allocation rules |
| Slow queries | Poor mappings, large shards, heavy aggregations | Query profile, shard size | Add filters, adjust mappings, reduce shard size |
| High heap usage | JVM pressure, large fields, heavy aggregations | Node stats, GC logs | Increase heap, optimize queries, reindex |
| Shard imbalance | Uneven distribution | `_cat/shards` | Trigger shard rebalance |
| Indexing failures | Mapping conflicts, full disk | `_cat/indices`, logs | Fix mappings, free disk, reindex |
| ILM stuck | Policy misconfig or rollover failure | `_ilm/explain` | Fix policy, force rollover |
