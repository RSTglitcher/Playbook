:::tip REST APIs  
REST APIs expose **resource‑based HTTP endpoints** that allow systems to communicate in a predictable, scalable, language‑agnostic way. They enable data exchange, service integration, and structured access to backend functionality.  
:::

### Key Terms

- **Resource:** A domain object exposed via an endpoint (e.g., `/users`, `/orders`)  
- **Endpoint:** A URL + HTTP method combination representing an operation  
- **Status Code:** Standardised HTTP response indicator (200, 400, 404, 500)  
- **Payload:** JSON body sent or received in a request  
- **Headers:** Metadata for auth, content type, caching, etc.  

### Knowing when to use **REST APIs**

| When to use | When not to use |
|-------------|------------------|
| Standard request–response communication | Real‑time bidirectional messaging (use WebSockets) |
| CRUD operations on resources | High‑throughput streaming (use gRPC or event streams) |
| Cross‑team integration | When strict schema enforcement is required (use GraphQL/OpenAPI‑first) |
| Public or internal service interfaces | When latency must be extremely low |

## 📘 About / Foundations

### How It Works (Conceptual Model)

REST APIs use **HTTP methods** (GET, POST, PUT, DELETE) to operate on **resources**.  
Clients send requests containing:

- URL  
- Method  
- Headers  
- Optional JSON body  

Servers return:

- Status code  
- Headers  
- Optional JSON body  

REST is stateless: each request contains all necessary context.

### System Contract

REST APIs define a contract through:

- **Endpoint structure**  
- **Request/response schemas**  
- **Status codes**  
- **Authentication rules**  
- **Versioning strategy**  
- **Error formats**  

Clients rely on this contract to integrate safely.

### Key Strategies

- Use **consistent naming** (`/users`, `/orders/{id}`)  
- Use **correct HTTP methods** (GET for read, POST for create)  
- Return **meaningful status codes**  
- Provide **clear error messages**  
- Version endpoints (`/v1/…`)  
- Enforce **authentication and authorization**  
- Log latency, payload size, and failures  

### What Results Normally Look Like

- JSON responses  
- Structured error objects  
- Status codes indicating success or failure  
- Logs showing request/response cycles  
- Metrics for latency, throughput, and error rates

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|---------|---------------------|---------------|-------------------|
| **400 Bad Request** | Invalid input | Body, params, schema | Fix payload or update validation |
| **401/403 Unauthorized** | Missing/invalid auth | Tokens, headers, roles | Refresh token, update permissions |
| **404 Not Found** | Wrong endpoint or version | URL, path params, version | Correct route or update spec |
| **429 Too Many Requests** | Rate limit exceeded | Rate limit headers | Add retries/backoff |
| **500 Internal Server Error** | Server failure | Logs, downstream dependencies | Escalate, fix integration |
| **503 Service Unavailable** | API overloaded or down | Health checks, gateway logs | Retry with backoff, escalate |
| **Slow responses** | Latency in downstream services | Tracing, logs, payload size | Optimize queries, reduce payload |
