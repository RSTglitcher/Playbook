:::tip MySQL  
MySQL is a widely used **relational database management system (RDBMS)** designed for structured data storage, transactional workloads, and SQL‑based querying.  
It powers applications ranging from internal tools to large‑scale production systems with replication, indexing, and high‑availability setups.  
:::

### Key Terms

- **Table:** Structured collection of rows and columns  
- **Primary Key:** Unique identifier for a row  
- **Foreign Key:** Relationship constraint between tables  
- **Index:** Data structure that speeds up queries  
- **Transaction:** Atomic unit of work (`BEGIN`, `COMMIT`, `ROLLBACK`)  
- **Storage Engine:** InnoDB (default), MyISAM, Memory  
- **Query Plan:** Execution strategy chosen by MySQL (`EXPLAIN`)  
- **Replication:** Copying data from primary to replicas  

### Knowing when to use **MySQL**

| When to use | When not to use |
|-------------|-----------------|
| Structured, relational data | Unstructured or schema‑less data |
| Strong consistency and ACID transactions | High‑throughput event ingestion |
| OLTP workloads | Large analytical workloads (use BigQuery/Snowflake) |
| Complex joins and constraints | Wide‑column or time‑series workloads |
| Mature SQL ecosystem | Highly distributed NoSQL architectures |

## 📘 About / Foundations  
_Conceptual grounding for how the platform works._

### How It Works (Conceptual Model)

- Data is stored in **tables** with rows and typed columns  
- **Indexes** accelerate lookups and joins  
- **Constraints** enforce data integrity  
- **Transactions** ensure atomicity and consistency  
- **Query optimizer** chooses execution plans  
- **Storage engines** determine durability and performance characteristics  
- **Replication** provides read scaling and failover  

### System Contract

MySQL enforces contracts through:

- **Schema definitions** (tables, columns, constraints)  
- **Data types** (INT, VARCHAR, DATETIME, JSON, etc.)  
- **Indexing strategy**  
- **Transaction isolation levels**  
- **User permissions and roles**  
- **Replication configuration**  
- **Backup and restore policies**  

### Key Strategies

<Tabs>
<TabItem value="schema" label="Schema Strategy">

- Use normalized schemas for OLTP workloads  
- Add foreign keys for referential integrity  
- Use appropriate data types (avoid oversized VARCHAR)  
- Responsibility: SWE + TA  
- Goal: maintainable, consistent data models  

</TabItem>

<TabItem value="performance" label="Performance Strategy">

- Use indexes for frequent filters and joins  
- Use `EXPLAIN` to inspect query plans  
- Avoid full table scans on large tables  
- Responsibility: SWE  
- Goal: predictable, efficient queries  

</TabItem>

<TabItem value="operations" label="Operations Strategy">

- Use replication for read scaling  
- Schedule regular backups  
- Monitor slow query logs  
- Responsibility: TA + Platform  
- Goal: reliable, recoverable database operations  

</TabItem>
</Tabs>

### What Results Normally Look Like

- Query results (rows, columns)  
- Schema metadata (`SHOW TABLES`, `DESCRIBE`)  
- Query plans (`EXPLAIN`)  
- Index usage statistics  
- Replication status  
- Backup files (`mysqldump`)

## 🚨 Triage

:::danger  
Symptoms and issues with MySQL and how to triage:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|---------|---------------------|---------------|-------------------|
| Slow queries | Missing indexes or inefficient SQL | `EXPLAIN`, table size, index usage | Add indexes, rewrite query |
| Locking / deadlocks | High contention or long transactions | `SHOW ENGINE INNODB STATUS` | Reduce transaction scope, add indexes |
| Connection failures | Network, credentials, or max connections | Logs, firewall, `max_connections` | Fix access, increase limits |
| Replication lag | Heavy writes or slow replica | Replica status, slow queries | Optimize writes, scale replicas |
| Data inconsistency | Missing constraints or ETL issues | Foreign keys, triggers, ETL jobs | Add constraints, fix ETL logic |
| Disk full | Large tables or logs | Disk usage, binlogs | Purge logs, archive data |
| High CPU usage | Expensive queries | Slow query log | Optimize queries, add indexes |
