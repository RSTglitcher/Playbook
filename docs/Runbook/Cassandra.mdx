:::tip Cassandra  
Apache Cassandra is a **distributed NoSQL database** designed for **high availability, horizontal scalability, and fault tolerance**.  
It handles large volumes of structured data across multiple nodes with no single point of failure.  
:::

### Key Terms

- **Node:** A Cassandra server participating in the cluster  
- **Data Center (DC):** Logical grouping of nodes for replication and locality  
- **Replication Factor (RF):** Number of copies of data stored across nodes  
- **Partition Key:** Determines data distribution across the cluster  
- **Consistency Level:** Read/write guarantee (e.g., ONE, QUORUM, ALL)  
- **SSTable:** Immutable on‑disk storage file  
- **Compaction:** Process of merging SSTables to reclaim space  
- **Tombstone:** Marker for deleted data, cleaned up during compaction  

### Knowing when to use **Cassandra**

| When to use | When not to use |
|-------------|-----------------|
| High‑write, high‑availability workloads | Workloads requiring strong ACID transactions |
| Multi‑region or multi‑DC replication | Complex joins or relational modelling |
| Predictable, low‑latency reads/writes | OLAP/analytics workloads |
| Horizontal scaling with no downtime | Small datasets where simpler DBs suffice |

## 📘 About / Foundations  
_Conceptual grounding for how the tool works._

### How It Works (Conceptual Model)

- Cassandra uses a **peer‑to‑peer architecture** — no master node.  
- Data is distributed using **consistent hashing** across nodes.  
- Replication is configured per keyspace using **NetworkTopologyStrategy**.  
- Writes are fast due to **log‑structured storage** (memtables + SSTables).  
- Reads merge data from memtables, caches, and SSTables.  
- Compaction merges SSTables and removes tombstones.  
- Repairs ensure consistency across replicas.

### System Contract

Cassandra enforces contracts through:

- **Keyspace replication strategy**  
- **Partition key design** (determines distribution and performance)  
- **Consistency levels** (read/write guarantees)  
- **Schema definition** (tables, clustering columns, TTLs)  
- **Node topology** (DCs, racks, seeds)

### Key Strategies

<Tabs>
<TabItem value="partitioning" label="Partitioning Strategy">

- Choose partition keys that avoid hotspots  
- Ensure even distribution across nodes  
- Avoid unbounded partitions (e.g., time‑series without bucketing)  
- Responsibility: SWE + TA validation  
- Goal: predictable performance and scalability  

</TabItem>

<TabItem value="replication" label="Replication & Durability Strategy">

- Use **NetworkTopologyStrategy** for multi‑DC setups  
- Set RF ≥ 3 for production workloads  
- Align consistency levels with RF (e.g., QUORUM reads/writes)  
- Responsibility: Platform team + TA oversight  
- Goal: durability, availability, and predictable consistency  

</TabItem>

<TabItem value="maintenance" label="Maintenance Strategy">

- Schedule regular repairs (incremental recommended)  
- Monitor compaction and tombstone accumulation  
- Ensure snapshot/backup processes are reliable  
- Responsibility: SWE + TA collaboration  
- Goal: cluster health and long‑term stability  

</TabItem>
</Tabs>

### What Results Normally Look Like

- `nodetool status` → node health, load, tokens, state  
- `DESCRIBE KEYSPACE` → replication, tables, schema  
- `nodetool compactionstats` → compaction backlog  
- `nodetool tpstats` → thread pool saturation  
- `nodetool cfstats` → SSTable counts, tombstones, read/write latency

## 🧩 Technical Skills

<details>
<summary>Pre‑Requisites</summary>

#### Security
- Credentials for `cqlsh`  
- Role‑based access for schema changes  
- Network access to nodes (ports 9042, 7199 for JMX)

#### Networking
- Node IPs and seed nodes  
- Data center and rack configuration  
- Firewall rules and VPC/subnet access  
- JMX connectivity for `nodetool`

#### Architecture
- Replication factor and keyspace strategy  
- Number of nodes per DC  
- Storage type (SSD recommended)  
- Compaction strategy (STCS, LCS, TWCS)

#### Data
- Keyspace and table schemas  
- Partition key and clustering column design  
- TTL usage and tombstone expectations  
- Backup and snapshot retention policies

#### Operational Guarantees
- Consistency levels (ONE, QUORUM, ALL)  
- Repair frequency and strategy  
- Compaction behaviour  
- Node failure tolerance

#### Observability Metrics
- Disk usage per node  
- SSTable counts  
- Read/write latency  
- Pending compactions  
- Tombstone warnings  
- Repair progress  

</details>

<TabItem value="engineering" label="Engineering">

Common workflows and commands used by engineers when operating this tool

```shell
--------------
Key Terminal Commands
--------------

# Connect to cluster
cqlsh <node_ip> -u <username> -p <password>

# Create keyspace
CREATE KEYSPACE my_keyspace
WITH REPLICATION = {'class': 'NetworkTopologyStrategy', 'dc1': 3};

# Create table
CREATE TABLE my_keyspace.users (
  id UUID PRIMARY KEY,
  name text,
  email text
);

# Insert data
INSERT INTO my_keyspace.users (id, name, email)
VALUES (uuid(), 'Alice', 'alice@example.com');

# Query data
SELECT * FROM my_keyspace.users;

# Delete data
DELETE FROM my_keyspace.users WHERE id=<uuid>;

# Scale cluster
nodetool bootstrap

# Trigger repairs
nodetool repair <keyspace_name>

# Take snapshots/backups
nodetool snapshot <keyspace_name>
```

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|--------|---------------------|---------------|-------------------|
| Node down | Hardware failure, network issue, JVM crash | `nodetool status`, system logs | Restart node, check network, rebalance cluster |
| Cluster inconsistency | Missed repairs, failed writes | Repair status, hinted handoff | Run repair, verify replication factor |
| Slow queries | High cardinality, unindexed queries | Query logs, schema, compaction stats | Optimize schema, add indexes, adjust compaction |
| High disk usage | SSTable accumulation, compaction backlog | `nodetool cfstats`, disk usage | Run compaction, clean snapshots, expand storage |
| Tombstone issues | Excessive deletes / TTL expiry | `nodetool cfstats`, read latency | Tune TTL, monitor tombstones, repair cluster |
| Backup failures | Misconfigured snapshot/backup process | Backup logs, snapshot status | Fix backup config, validate permissions |
