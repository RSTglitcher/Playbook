:::tip GraphQL
GraphQL is a query language and runtime for APIs that allows clients to request exactly the data they need. It enables efficient, strongly‑typed data retrieval across complex systems by exposing a unified schema of types, fields, and resolvers.
:::

<TabItem value="requirements" label="Requirements">

**How to formulate requirements when working with this platform**

- Clarify the domain model and how it maps to GraphQL types
- Identify required fields, optional fields, and validation rules
- Confirm expected query shapes and nested relationships
- Understand resolver dependencies and data sources
- Capture error handling expectations and nullability rules
- Define performance expectations (depth, complexity, batching)

**Examples of use cases**

- Designing new queries or mutations for a feature
- Validating schema changes before engineering implementation
- Supporting frontend teams with data‑shape requirements
- Mapping business logic to GraphQL types and relationships

**What a TA needs to gather**

- Schema documentation (SDL, introspection, or tooling)
- Query/mutation examples and expected outputs
- Data mapping rules and transformation logic
- Authentication and authorization requirements
- Edge cases, nullability rules, and error scenarios

**How to formulate a plan for development**

- Define new types, fields, and input objects
- Identify resolver logic and downstream dependencies
- Plan for batching, caching, and avoiding N+1 issues
- Establish error handling and nullability rules
- Determine authorization strategy at field or resolver level

**Examples of implementation**

- Adding a new mutation for a feature
- Extending an existing type with new fields
- Creating resolvers that aggregate multiple services
- Implementing subscriptions for real‑time updates

**Engineering considerations**

- Schema evolution and backward compatibility
- Field‑level authorization
- Query complexity limits and depth restrictions
- Resolver performance and caching strategy
- Consistent error shapes and nullability

**What a TA needs to provide for handover**

- Clear definitions of types, fields, and relationships
- Business rules for mutations and validation
- Expected query shapes and nested data requirements
- Edge cases, nullability expectations, and error scenarios
- Dependencies and integration points

## Triage

:::danger
Symptoms and issues with this platform and how to triage this:
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|---------|---------------------|---------------|-------------------|
| Slow queries | N+1 issues or deep nested queries | Resolver logs, query depth | Add batching/caching, limit depth |
| Null fields | Resolver failure or missing data | Downstream services, schema nullability | Fix resolver or update schema |
| Authorization errors | Missing or invalid tokens | Headers, roles, field-level auth | Refresh token or update permissions |
| Schema mismatch | Outdated schema or breaking change | SDL, introspection, versioning | Sync schema or update clients |
| Unexpected errors | Resolver exceptions | Logs, downstream dependencies | Fix resolver logic or escalate |
