:::tip GraphQL  
GraphQL is a **query language and runtime** for APIs that allows clients to request exactly the data they need.  
It enables efficient, strongly‑typed data retrieval across complex systems by exposing a unified schema of types, fields, and resolvers.  
:::

### Key Terms

- **Schema:** The contract defining types, fields, and relationships  
- **Query:** Read operation requesting specific fields  
- **Mutation:** Write operation modifying data  
- **Subscription:** Real‑time event stream  
- **Resolver:** Function that returns data for a field  
- **Type:** Object, scalar, enum, interface, union  
- **Input Type:** Structured input for mutations  
- **Directive:** Annotation modifying execution (`@include`, `@skip`)  
- **Introspection:** Ability to query the schema itself  

### Knowing when to use **GraphQL**

| When to use | When not to use |
|-------------|-----------------|
| Complex client apps needing flexible data | Simple CRUD APIs |
| Multiple clients with different data needs | High‑throughput streaming (Kafka, Pub/Sub) |
| Strongly‑typed API contracts | Large binary uploads |
| Avoiding over‑fetching/under‑fetching | Simple internal APIs where REST is sufficient |
| Aggregating multiple backends | Event‑driven workflows |

## 📘 About / Foundations  
_Conceptual grounding for how the tool works._

### How It Works (Conceptual Model)

- Clients send **queries** specifying exactly which fields they want  
- The **schema** defines types, fields, and relationships  
- **Resolvers** fetch data for each field, often from downstream services  
- Execution walks the query as a **tree**, resolving fields recursively  
- Mutations modify data and return updated objects  
- Subscriptions (if enabled) push real‑time updates  
- Introspection exposes the schema for tooling and validation  

### System Contract

GraphQL enforces contracts through:

- **Schema Definition Language (SDL)**  
- **Type system** (objects, scalars, enums, interfaces, unions)  
- **Field‑level resolver functions**  
- **Validation rules** (queries must match schema)  
- **Nullability rules** (non‑null fields must always resolve)  
- **Schema evolution strategy** (prefer additive changes)  
- **Query complexity and depth limits**  

### Key Strategies

<Tabs>
<TabItem value="schema" label="Schema Strategy">

- Keep types small and composable  
- Avoid deeply nested structures  
- Use interfaces/unions for polymorphism  
- Prefer cursor‑based pagination  
- Responsibility: SWE + TA validation  
- Goal: stable, evolvable API contracts  

</TabItem>

<TabItem value="performance" label="Performance Strategy">

- Use batching (DataLoader) to avoid N+1 queries  
- Cache expensive resolvers  
- Apply query depth and complexity limits  
- Responsibility: SWE  
- Goal: predictable, efficient execution  

</TabItem>

<TabItem value="versioning" label="Versioning Strategy">

- Avoid breaking changes  
- Deprecate fields instead of removing them  
- Use schema federation for large systems  
- Responsibility: SWE + TA  
- Goal: safe evolution of API surface  

</TabItem>
</Tabs>

### What Results Normally Look Like

- JSON responses with exactly the requested fields  
- Schema introspection results  
- Resolver logs and performance traces  
- Query plans (depending on server)  
- Error objects with `message`, `path`, and `extensions`

- Clarify the domain model and how it maps to GraphQL types  
- Identify required vs optional fields  
- Confirm expected query shapes and nested relationships  
- Understand resolver dependencies and data sources  
- Capture error handling expectations and nullability rules

- Schema documentation (SDL or introspection)  
- Query/mutation examples and expected outputs  
- Data mapping rules and transformation logic  
- Authentication and authorization requirements

Key Terminal Commands

# Defining schema types
type User {
  id: ID!
  name: String!
  email: String
}

# Implementing resolvers
const resolvers = {
  Query: {
    user: (_, { id }, { dataSources }) =>
      dataSources.usersAPI.getUser(id)
  }

# Running mutations
mutation {
  createUser(input: { name: "Alice" }) {
    id
    name
  }

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|---------|---------------------|---------------|-------------------|
| Slow queries | N+1 issues or deep nested queries | Resolver logs, query depth | Add batching/caching, limit depth |
| Null fields | Resolver failure or missing data | Downstream services, schema nullability | Fix resolver or update schema |
| Authorization errors | Missing or invalid tokens | Headers, roles, field‑level auth | Refresh token or update permissions |
| Schema mismatch | Outdated schema or breaking change | SDL, introspection, versioning | Sync schema or update clients |
| Unexpected errors | Resolver exceptions | Logs, downstream dependencies | Fix resolver logic or escalate |
