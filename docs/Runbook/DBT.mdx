:::tip dbt  
dbt (data build tool) is a **data transformation framework** that enables analysts and engineers to build modular SQL models, enforce data quality tests, manage dependencies, and generate documentation — all version‑controlled and reproducible.  
It solves problems related to **transformations, testing, lineage, CI/CD, and analytics engineering workflows**.  
:::

### Key Terms

- **Model:** A SQL transformation compiled and run by dbt  
- **Source:** External raw data referenced by dbt  
- **Seed:** CSV file managed and loaded by dbt  
- **Macro:** Reusable Jinja logic for SQL generation  
- **DAG:** Directed Acyclic Graph representing model dependencies  
- **Tests:** Assertions on data quality (unique, not null, accepted values)  
- **Profiles.yml:** Connection configuration for environments  

### Knowing when to use **dbt**

| When to use | When not to use |
|-------------|-----------------|
| SQL‑based transformations in a warehouse | Heavy ETL/ELT requiring Python or Spark |
| You need lineage, testing, and modular SQL | Real‑time streaming transformations |
| You want version‑controlled analytics code | Operational pipelines requiring orchestration |
| You need CI/CD for data models | Complex ML pipelines or non‑SQL workloads |

## 📘 About / Foundations  
_Conceptual grounding for how the tool works._

### How It Works (Conceptual Model)

- dbt compiles **modular SQL files** into executable SQL for the warehouse  
- Models form a **DAG**, ensuring correct execution order  
- Tests validate data quality at model or column level  
- Documentation is auto‑generated from model metadata  
- Environments (dev/stage/prod) are configured via `profiles.yml`  
- dbt Cloud or CI pipelines automate runs and testing  

### System Contract

dbt enforces contracts through:

- **Model definitions** (`.sql` files)  
- **Schema.yml files** defining tests, descriptions, and metadata  
- **Source freshness** checks  
- **Tags** and **selectors** controlling execution scope  
- **Profiles.yml** defining warehouse credentials and environment behaviour  

### Key Strategies

<Tabs>
<TabItem value="modelling" label="Modelling Strategy">

- Use **staging → intermediate → mart** layers  
- Keep models modular and composable  
- Use CTEs sparingly; prefer model references (`ref()`)  
- Responsibility: SWE + TA validation  
- Goal: maintainable, testable, lineage‑driven SQL  

</TabItem>

<TabItem value="testing" label="Testing Strategy">

- Apply **unique**, **not null**, and **accepted values** tests  
- Add custom tests for business logic  
- Monitor test failures in CI  
- Responsibility: Analysts + TAs  
- Goal: reliable, trustworthy data models  

</TabItem>

<TabItem value="performance" label="Performance Strategy">

- Use incremental models for large tables  
- Partition and cluster warehouse tables where possible  
- Monitor warehouse cost and query performance  
- Responsibility: SWE + TA collaboration  
- Goal: efficient, scalable transformations  

</TabItem>
</Tabs>

### What Results Normally Look Like

- `dbt run` → executed models with pass/fail status  
- `dbt test` → test results with failures highlighted  
- `dbt docs generate` → documentation site with lineage graph  
- CI logs showing model execution order and failures  
- Source freshness reports

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|--------|---------------------|---------------|-------------------|
| Models failing | SQL errors, missing tables, broken references | `dbt run` logs, warehouse schema | Fix SQL, update references, rerun |
| Tests failing | Data quality or constraints violated | `dbt test` output, model source data | Correct data or model logic, rerun tests |
| DAG dependency broken | Model dependency misconfigured or deleted | DAG graph, `dbt ls` | Update dependencies, verify DAG, rerun |
| Documentation not updating | Docs not generated or build failed | `dbt docs` logs, artifacts | Regenerate docs and serve |
| CI/CD pipeline failing | Git issues, environment misconfig | CI logs, profiles.yml | Fix CI config, validate environment |
| Slow runs / warehouse overload | Large incremental loads, high cardinality | dbt run timing, warehouse metrics | Optimize queries, shard incremental runs, adjust warehouse size |
