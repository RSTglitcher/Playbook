:::tip Prometheus  
Prometheus is a **metrics collection and monitoring system** designed for reliability, dimensional data, and powerful querying via PromQL.  
It solves problems related to **time‑series metrics, alerting, service monitoring, and SRE observability**.  
:::

### Key Terms

- **Metric:** A time‑series identified by a name and labels  
- **Label:** Key/value metadata attached to metrics  
- **Scrape:** Prometheus pulling metrics from a target  
- **Target:** An endpoint exposing metrics (e.g., `/metrics`)  
- **Exporter:** A service that exposes metrics in Prometheus format  
- **PromQL:** Query language for Prometheus  
- **Alertmanager:** Component that routes alerts to Slack, PagerDuty, email  
- **Retention:** How long metrics are stored locally  

### Knowing when to use **Prometheus**

| When to use | When not to use |
|-------------|-----------------|
| Service‑level metrics (CPU, latency, errors, throughput) | Log search (use Loki/Kibana) |
| Alerting on thresholds or SLOs | Long‑term analytics (use BigQuery/Snowflake) |
| Kubernetes cluster monitoring | High‑cardinality event data |
| Dimensional metrics with labels | Distributed tracing (use Jaeger/Tempo) |
| SRE/SLO monitoring | Business dashboards (use Looker/PowerBI) |

## 📘 About / Foundations  
_Conceptual grounding for how the tool works._

### How It Works (Conceptual Model)

- Prometheus **scrapes** metrics from targets at fixed intervals  
- Metrics are stored locally as **time‑series**  
- PromQL queries aggregate, filter, and transform metrics  
- Alert rules evaluate PromQL expressions  
- Alertmanager handles routing, grouping, and silencing  
- Exporters expose metrics for systems that don’t natively support Prometheus  
- Service discovery (Kubernetes, EC2, Consul) dynamically updates targets  

### System Contract

Prometheus enforces contracts through:

- **Scrape configs** (targets, intervals, relabeling)  
- **Metric naming conventions** (`http_requests_total`)  
- **Label cardinality constraints**  
- **Alerting rules** (PromQL expressions + conditions)  
- **Retention policies**  
- **Exporter formats** (OpenMetrics)  

### Key Strategies

<Tabs>
<TabItem value="metrics" label="Metrics Strategy">

- Use consistent naming (`<component>_<metric>_<unit>`)  
- Avoid high‑cardinality labels (user IDs, request IDs)  
- Use histograms for latency and distribution metrics  
- Responsibility: SWE + TA  
- Goal: reliable, queryable metrics  

</TabItem>

<TabItem value="scraping" label="Scraping Strategy">

- Use Kubernetes service discovery where possible  
- Tune scrape intervals based on SLOs  
- Use relabeling to reduce noise  
- Responsibility: Platform + SWE  
- Goal: efficient, scalable scraping  

</TabItem>

<TabItem value="alerting" label="Alerting Strategy">

- Use multi‑condition alerts (threshold + duration)  
- Group alerts by service and severity  
- Use silence windows for deployments  
- Responsibility: TA + SRE  
- Goal: actionable, low‑noise alerting  

</TabItem>
</Tabs>

### What Results Normally Look Like

- PromQL query results (time‑series, vectors)  
- Target health (`UP`/`DOWN`)  
- Alert firing/resolved events  
- Exporter metrics (node_exporter, kube-state-metrics)  
- Rule evaluation logs  
- Retention and storage usage

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|--------|---------------------|---------------|-------------------|
| Target DOWN | Network issue or exporter crash | `/targets`, exporter logs | Fix exporter, restore connectivity |
| High cardinality | Unbounded labels | Label schema, metric names | Remove high‑cardinality labels |
| Slow queries | Heavy PromQL or large datasets | Query inspector, Prometheus logs | Optimize queries, add recording rules |
| Alerts not firing | Wrong condition or data source down | Alert rules, Alertmanager logs | Fix rule, restore data source |
| Prometheus OOM | Too many metrics or high scrape frequency | TSDB stats, WAL size | Reduce retention, adjust scrape interval |
| Missing metrics | Exporter misconfig or scrape failure | `/targets`, exporter endpoint | Fix exporter, update scrape config |
| Alert spam | Overly sensitive thresholds | Alert rules, firing frequency | Tune thresholds, add duration windows |

## 📘 PromQL Cheat Sheet (Analyst‑Focused)

Below is a compact cheat sheet you can embed directly under the Analyst tab or keep as a reference block.

### Error & Traffic Patterns

| Goal | PromQL |
|------|--------|
| Error rate (5xx) | `rate(http_requests_total{status=~"5.."}[5m])` |
| Traffic volume | `rate(http_requests_total[5m])` |
| Requests per instance | `sum by(instance)(rate(http_requests_total[5m]))` |

### Latency (Histograms)

| Goal | PromQL |
|------|--------|
| p95 latency | `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))` |
| p99 latency | `histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))` |
| Latency per endpoint | `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket{handler!=""}[5m]))` |

### Resource Usage

| Goal | PromQL |
|------|--------|
| CPU usage | `rate(node_cpu_seconds_total{mode!="idle"}[5m])` |
| Memory available | `node_memory_MemAvailable_bytes` |
| Disk usage | `node_filesystem_avail_bytes` |

### Instance & Service Health

| Goal | PromQL |
|------|--------|
| Count healthy instances | `count(up == 1)` |
| Identify down instances | `up == 0` |
| Restart count | `changes(process_start_time_seconds[1h])` |

### Detecting Spikes, Drops, and Trends

| Goal | PromQL |
|------|--------|
| Sudden spike | `deriv(metric_name[10m]) > threshold` |
| Sudden drop | `deriv(metric_name[10m]) < -threshold` |
| Rolling average | `avg_over_time(metric_name[5m])` |

### High‑Cardinality Detection

| Goal | PromQL |
|------|--------|
| Top 10 metrics by label count | `topk(10, count by(__name__)({__name__!=""}))` |
| High‑cardinality label values | `count by(label)(metric_name)` |
