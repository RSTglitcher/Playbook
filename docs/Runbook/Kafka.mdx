:::tip Kafka  
**Apache Kafka** is a distributed event streaming platform used to ingest, store, and process high‑volume real‑time data using an append‑only log. It enables scalable, fault‑tolerant, real‑time pipelines and event‑driven architectures.  
:::

### Key Terms

- **Topic:** A named category/feed where messages are written  
- **Partition:** A shard of a topic; each is an independent append‑only log  
- **Offset:** A monotonically increasing identifier for each record  
- **Producer:** Writes messages to Kafka  
- **Consumer:** Reads messages from Kafka  
- **Broker:** A Kafka server; multiple brokers form a cluster  
- **Consumer Group:** A set of consumers sharing work across partitions  
- **Retention:** How long Kafka keeps data (time or size based)  
- **Fan‑out:** Many consumers reading the same data independently  

### Knowing when to use **Kafka**

| When to use | When not to use |
|-------------|-----------------|
| Real‑time streaming between systems | Request/response workflows (use REST/gRPC) |
| High‑throughput ingestion (MB/s → GB/s) | Complex routing/transformations (use Connect/ESB) |
| Event‑driven architecture | Long‑term storage or warehousing |
| Durable logs with replay | Low‑volume workloads where simpler queues suffice |
| Decoupling producers and consumers | Systems requiring strict global ordering |

## 📘 About / Foundations  
_Conceptual grounding for how the tool works._

### How It Works (Conceptual Model)

- Kafka is a **distributed append‑only log**.  
- Producers publish events to **topics**, which are split into **partitions**.  
- Consumers read events sequentially using **offsets**.  
- Brokers persist logs and replicate partitions for durability.  
- Ordering is guaranteed **within a partition**, not across partitions.  
- Kafka uses its own **binary protocol over TCP**, not HTTP.  

### System Contract

Kafka stores **raw bytes**, not schemas.  
Schema contracts may exist in:

- Application code  
- YAML configuration  
- Protobuf / Avro definitions  
- Schema Registry  
- Sample files

<Tabs>

- Choose partition keys deliberately  
- Ensure even distribution to avoid hot partitions  
- Align partitioning with consumer scaling needs  
- Responsibility: SWE + TA validation  
- Goal: throughput, parallelism, predictable ordering  

</TabItem>

<TabItem value="retention" label="Retention & Durability Strategy">

- Define retention by time or size  
- Ensure compliance with governance (PII, audit)  
- Validate storage capacity and broker configuration  
- Responsibility: Platform team + TA oversight  
- Goal: balance durability, cost, performance  

</TabItem>
</Tabs>

### What Results Normally Look Like

- Streams of events (key/value byte records)  
- Offsets indicating position in the log  
- Consumer lag metrics (MAE, MCE)  
- Partition‑level throughput statistics  

Notes:  
Kafka stores **raw bytes**. Serialization is handled by clients.  
Ordering is guaranteed **within a partition**, not across partitions.

## 🧩 Technical Skills

<details>
<summary>Pre‑Requisites</summary>

#### Security
- Authentication method (TLS, mTLS, SASL)  
- Certificate issuance + keystore/truststore management  
- ACLs and topic‑level permissions  
- Schema Registry access (if used)

#### Networking
- Broker reachability (firewalls, VPCs, proxies, DNS)  
- Bootstrap server stability  
- Cross‑region latency expectations  
- Kafka protocol (TCP) connectivity

#### Architecture
- Cluster topology (brokers, replication factor, partitions)  
- Retention policies (time/size/compaction)  
- KRaft vs Zookeeper  
- Expected throughput and message size

#### Data
- Event schema definition  
- Schema evolution strategy  
- Topic naming conventions  
- Domain boundaries for each topic

#### Operational Guarantees
- Delivery semantics (at‑least‑once, at‑most‑once, exactly‑once)  
- Ordering requirements  
- Retry/backoff strategy  
- DLQ or equivalent pattern

#### Observability Metrics
- Consumer lag (MAE/MCE)  
- Broker health (under‑replicated partitions, offline partitions)  
- Throughput metrics  
- Retention and disk usage  
- Alerting thresholds  

</details>

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|--------|---------------------|---------------|-------------------|
| Brokers down | Node failure, JVM crash, network issue | Broker logs, cluster status, controller broker | Restart broker, fix network, rebalance partitions |
| Consumer lag | Slow consumers, blocked processing | Offsets, MAE/MCE, partition skew | Add consumers, fix bottlenecks, rebalance |
| Topic unavailable | Deleted, ACL issue, misconfiguration | Topic list, ACLs, logs | Recreate topic, fix ACLs |
| Messages not delivered | Retention expired, partition offline | Retention, partition status | Increase retention, fix partitions |
| High latency | Overloaded broker, network bottleneck | CPU, network throughput | Scale brokers, optimize network |
| Data loss | Under‑replicated partitions | RF, ISR list | Increase RF, restore brokers |
| Hot partitions | Poor partition key | Partition message counts | Redesign key, increase partitions |
| Serialization errors | Schema mismatch | Producer/consumer logs | Fix schema, enforce registry |
| High disk usage | Retention too long | Topic size, segment count | Adjust retention, compact logs |
| Slow startup / rebalance | Too many partitions | Logs, rebalance events | Reduce partitions, optimize group size |
