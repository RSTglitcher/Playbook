:::tip AWS Cloud Platform  
AWS provides a broad suite of **cloud services** including compute, storage, database, messaging, and analytics.  
This runbook focuses on key data and messaging services commonly used by engineers and monitored by TAs: **ElastiCache, DynamoDB, Keyspaces, SQS, and SNS**.  
:::

### Key Terms

- **IAM Role:** Identity with permissions to access AWS services  
- **VPC:** Virtual network boundary for AWS resources  
- **Security Group:** Virtual firewall controlling inbound/outbound traffic  
- **Region / AZ:** Physical locations where AWS resources run  
- **Endpoint:** Network entry point for a service (e.g., DynamoDB, Keyspaces)  

### Knowing when to use **AWS Cloud Platform**

| When to use | When not to use |
|-------------|-----------------|
| You need managed, scalable cloud services | You require full control over hardware or OS |
| You need serverless or autoscaling data systems | You need strict on‑prem or air‑gapped environments |
| You want managed messaging (SQS/SNS) | You need complex routing (use Kafka/ESB) |
| You need low‑ops NoSQL storage (DynamoDB/Keyspaces) | You need relational joins or OLAP workloads |

## 📘 About / Foundations  
_Conceptual grounding for how the tool works._

### How It Works (Conceptual Model)

AWS provides **managed services** that abstract away infrastructure tasks such as scaling, patching, failover, and replication.  
Key concepts:

- **DynamoDB** → serverless NoSQL key‑value store  
- **Keyspaces** → managed Apache Cassandra  
- **ElastiCache** → managed Redis/Memcached  
- **SQS** → distributed message queue  
- **SNS** → pub/sub notification service  

AWS services rely heavily on:

- **IAM** for access control  
- **VPC networking** for connectivity  
- **CloudWatch** for metrics and logs  
- **Endpoints** for regional access  

### System Contract

AWS services enforce contracts through:

- **IAM policies** (who can do what)  
- **Resource configuration** (tables, clusters, queues, topics)  
- **Service‑specific APIs** (DynamoDB API, Redis protocol, SQS/SNS APIs)  
- **Regional boundaries** (resources exist in a specific region)  

### Key Strategies

- Ensure IAM roles are scoped correctly (least privilege)  
- Validate VPC, subnets, and security groups before debugging service issues  
- Confirm service quotas (DynamoDB RCU/WCU, SQS throughput, SNS fan‑out)  
- Use CloudWatch metrics for triage and performance analysis  
- Ensure resources exist in the correct region and account  

### What Results Normally Look Like

- **DynamoDB:** table descriptions, throughput metrics, item counts  
- **Keyspaces:** keyspace/table metadata, CQL query results  
- **ElastiCache:** cluster/node status, memory usage, replication state  
- **SQS:** queue attributes, message counts, visibility timeout  
- **SNS:** topic ARNs, subscription lists, delivery status

## 🧩 Technical Skills

<details>
<summary>Pre‑Requisites</summary>

#### Security
- IAM roles and policies for DynamoDB, Keyspaces, ElastiCache, SQS, SNS  
- MFA or SSO access to AWS console  
- KMS encryption policies (if applicable)

#### Networking
- VPC, subnets, route tables  
- Security groups and NACLs  
- VPC endpoints for DynamoDB/Keyspaces/SQS/SNS  
- Redis port access for ElastiCache

#### Architecture
- Region and AZ selection  
- Table/cluster/queue/topic naming conventions  
- Scaling configuration (autoscaling, provisioned capacity)  
- Replication and failover strategy

#### Data
- DynamoDB partition/sort key design  
- Keyspaces schema (Cassandra model)  
- Redis key patterns  
- Message formats for SQS/SNS

#### Operational Guarantees
- DynamoDB consistency model (eventual vs strong)  
- SQS delivery semantics (at‑least‑once)  
- SNS fan‑out behaviour  
- ElastiCache replication guarantees  
- Keyspaces consistency levels

#### Observability Metrics
- CloudWatch metrics for each service  
- DynamoDB RCU/WCU consumption  
- SQS queue depth and age of oldest message  
- SNS delivery failures  
- ElastiCache CPU/memory/evictions  
- Keyspaces read/write latency

## 🚨 Triage

:::danger  
Symptoms and issues with this platform and how to triage this:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|--------|---------------------|---------------|-------------------|
| ElastiCache cluster down | Node crash, network issue, scaling limit | Cluster status, node events, VPC connectivity | Restart node/cluster, check limits |
| DynamoDB table unavailable | Provisioned throughput exceeded or table deleted | Table status, CloudWatch metrics | Scale table, recreate if missing |
| Keyspaces not reachable | VPC/Security issue or endpoint misconfigured | Describe keyspace, network configuration | Fix network/security, restart endpoints |
| SQS messages delayed/lost | Consumer down or misconfigured queue | Queue attributes, CloudWatch metrics | Restart consumer, check permissions |
| SNS notifications not sent | Topic misconfigured, IAM or throttling | Topic subscriptions, CloudWatch metrics | Correct config, adjust IAM, retry |
| IAM access denied | Missing permissions | IAM roles and policies | Grant required roles |
| API quota exceeded | Too many requests | Service quotas, CloudWatch logs | Request increase or throttle usage |
