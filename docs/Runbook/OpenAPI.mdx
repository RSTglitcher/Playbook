:::tip OpenAPI  
OpenAPI is a **standard API contract specification** used to formally define HTTP APIs so they can be validated, documented, published, and consumed consistently across teams and tooling.  
It enables design‑first development, governance, automated documentation, client generation, and reliable API evolution.  
:::

### Key Terms

- **Paths:** Endpoints and operations (`GET`, `POST`, etc.)  
- **Schemas:** Data models for requests and responses  
- **Components:** Reusable definitions (schemas, parameters, responses)  
- **Security Schemes:** Auth mechanisms (API keys, OAuth2, JWT)  
- **Servers:** Base URLs for environments  
- **Info Block:** Title, version, contact, description  
- **Spec Format:** YAML or JSON  
- **Versioning:** Semantic versioning for API lifecycle  

### Knowing when to use **OpenAPI**

| When to use | When not to use |
|-------------|-----------------|
| API contract governance | Internal-only APIs with no consumers |
| Design‑first API development | Event‑driven systems (Kafka, Pub/Sub) |
| Automated documentation | Ad‑hoc or experimental endpoints |
| Client SDK generation | Highly dynamic or schema‑less APIs |
| API linting and validation | RPC‑style APIs without HTTP semantics |

## 📘 About / Foundations  
_Conceptual grounding for how the platform works._

### How It Works (Conceptual Model)

- OpenAPI defines an API contract in **YAML or JSON**  
- The contract describes **paths**, **operations**, **schemas**, and **security**  
- Tools validate the spec for correctness and completeness  
- CI pipelines enforce linting and governance rules  
- Documentation portals (e.g., Backstage) render the spec for consumers  
- Code generators produce **clients**, **servers**, and **mocks**  
- Versioning ensures safe evolution of the API surface  

### System Contract

OpenAPI enforces contracts through:

- **Schema definitions** (types, required fields, enums)  
- **Operation definitions** (parameters, responses, status codes)  
- **Security schemes** (OAuth2, API keys, JWT)  
- **Versioning rules** (semantic versioning)  
- **Governance policies** (linting, naming conventions)  
- **Publishing pipelines** (Backstage, portals, repos)

## 🚨 Triage

:::danger  
Symptoms and issues with OpenAPI and how to triage:  
:::

| Symptom | TLDR of Root Cause | What to Check | Action to Propose |
|---------|---------------------|---------------|-------------------|
| Consumers report incorrect behaviour | Spec drift from implementation | Compare spec to live API responses | Update spec or raise defect |
| Backstage docs missing or broken | Invalid or unreachable spec | Check spec URL, format, publishing pipeline | Fix spec or pipeline |
| CI pipeline failing | Lint or validation errors | Review validator output | Fix schema or adjust rules |
| Breaking changes reported | Poor versioning or governance | Check version history | Introduce proper versioning |
| Security confusion | Missing or unclear auth definitions | Review security schemes | Clarify and standardise |
