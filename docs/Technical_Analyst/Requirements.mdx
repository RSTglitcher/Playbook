:::tip Concept Overview
A **Requirement** is a clear, testable statement that defines what a solution must do, support, or enable.
For a Technical Analyst, requirements are alignment tools — they reduce ambiguity, set boundaries, and ensure decisions are grounded in real constraints rather than assumptions or preferences.
- Translate business needs into actionable, testable conditions
- Establish boundaries for design and implementation
- Prevent scope creep and misalignment
- Provide a shared reference point for engineering, stakeholders, and QA
:::

---
### Principles

- **Clarity** – No ambiguity or interpretation gaps
- **Testability** – You can verify whether it is met
- **Neutrality** – No implied solution or emotional framing
- **Traceability** – Linked to a problem, risk, or constraint
- **Bounded Scope** – Defines limits, not open‑ended desires
- **Negotiability** – Explicitly states whether it is fixed or flexible

### Remit

- Capturing constraints that shape the solution space
- Documenting assumptions, dependencies, and non‑negotiables
- Ensuring requirements reflect real system, business, or operational needs
- Providing a structured foundation for design, estimation, and acceptance
- Acting as the connective tissue between **Problem → Requirements → Acceptance**

## Key Strategies

<Tabs>
<TabItem value="approach" label="Requirements Approach">

1. **Start with the problem** – Requirements must trace back to a real issue
2. **Identify constraints** – Technical, operational, regulatory, or business
3. **Write the requirement** – Clear, testable, and neutral
4. **Define acceptance criteria** – How we know it’s met
5. **Validate with stakeholders** – Ensure feasibility and alignment

</TabItem>

<TabItem value="scenarios" label="Scenario Analysis">

- Identify edge cases that influence requirement scope
- Consider environmental differences (dev/staging/prod)
- Evaluate performance, concurrency, and scaling needs
- Model failure modes and operational constraints

</TabItem>

<TabItem value="stakeholder" label="Stakeholder Alignment">

- Ensure all parties agree on what “done” means
- Highlight trade-offs and constraints early
- Document negotiable vs non‑negotiable items
- Provide clear, solution‑agnostic framing

</TabItem>
</Tabs>

## The TA Requirement Pattern

Every requirement should follow this structure:

1. **Context**
> Why this requirement exists — the underlying problem or constraint.

2. **Constraint**
> What limits the solution space (technical, operational, regulatory).

3. **Requirement Statement**
> A single sentence, structured as:
```
The solution must <do/enable/support> <behaviour or condition> because <reason>.
```

4. **Acceptance Criteria**
> How we verify the requirement is met.

5. **Negotiability**
> - **Non‑negotiable** (fixed constraint)
> - **Negotiable** (preference)
> - **Team‑specific** (must be supported via extension points)

## 🧩 Example Requirements (Side‑by‑Side)

<Tabs>

<TabItem value="good" label="Good Requirement">

```
Context:
The team deploys Python services to Kubernetes using a shared base image.

Constraint:
The base image is maintained by the platform team and cannot be modified.

Requirement:
The cookiecutter template must support using the shared Python base image because teams cannot maintain custom images.

Acceptance Criteria:
- Dockerfile references the shared base image
- No custom OS-level dependencies required
- Build pipeline succeeds using the shared image

Negotiability:
Non‑negotiable
```

</TabItem>

<TabItem value="bad" label="Bad Requirement">

```
We should use the platform image because that’s what everyone else does.
```

</TabItem>

</Tabs>

## Requirements Presentation Structure

1. **Executive Summary**
- What you analysed
- What you found
- What decisions are needed

2. **Project Context**
- Team composition
- Tech stack
- Constraints
- Deployment model

3. **Fixed vs Flexible vs Team‑Specific**
A table like this:

| Category | Items | Notes |
|---------|-------|-------|
| Fixed | Python 3.12, shared base image | Cannot change |
| Flexible | Test framework | Can change with effort |
| Team‑specific | ETL folder structure | Must be supported |

4. **Requirements List**
Each requirement written using the TA pattern.

5. **Adoption Conditions**
What must be true for the team to adopt the solution.

6. **Risks & Dependencies**
What could block or delay adoption.

7. **Next Steps**
Clear, shared actions for alignment and execution.
