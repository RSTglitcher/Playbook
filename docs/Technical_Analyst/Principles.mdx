:::tip Concept Overview
**Engineering Principles** are foundational guidelines that shape how software and data systems are designed, evolved, and operated.  
For a Technical Analyst, principles provide a lens for evaluating decisions, identifying risks, and understanding why certain trade-offs exist.
- Provide a shared vocabulary for reasoning about design and behaviour
- Help TAs evaluate systems without relying on specific tools or technologies
- Reveal the intent behind architectural decisions
- Prevent misuse of “best practices” by grounding them in context
- Support consistent, maintainable, and scalable solutions
:::

---
### Principles:

- **Contextual Application** – Principles are guides, not universal rules  
- **Trade‑off Awareness** – Every principle has a cost when applied blindly  
- **Neutrality** – Principles describe behaviour, not preferences  
- **Clarity** – Principles should simplify decision-making  
- **Traceability** – Principles should link to real constraints or risks  

### Remit:

- Understanding how principles apply across domains (SWE, Data, Platform)  
- Identifying when a principle is being violated or misapplied  
- Communicating trade-offs using principle‑based reasoning  
- Supporting design reviews, incident analysis, and requirement framing  
- Helping teams avoid over‑engineering or accidental complexity

## Cross‑Cutting Engineering Principles

These principles apply across Software Engineering, Data Engineering, and Platform work, but manifest differently depending on context.

<Tabs>
<TabItem value="dry" label="DRY">

### Don’t Repeat Yourself (DRY)
A system should have a single source of truth for *logic, rules, or knowledge*.

**In Software Engineering**
- Strongly enforced  
- Reduces bugs and inconsistent behaviour  
- Duplication usually indicates poor abstraction  

Good DRY target: business logic, algorithms, rules

**In Data Engineering**
- Applied selectively  
- Logic should be DRY, data often should not be  
- Duplication may be intentional for:  
  - Performance  
  - Historical accuracy  
  - Auditability  
  - Isolation  

**TA rule of thumb:**  
👉 *DRY the logic, duplicate the facts when needed.*

</TabItem>

<TabItem value="kiss" label="KISS">

### Keep It Simple, Stupid (KISS)
Prefer the simplest solution that meets the requirements.

**SWE perspective**
- Avoid unnecessary abstractions  
- Reduce cognitive load  
- Improve maintainability  

**Data perspective**
- Prefer understandable pipelines over clever ones  
- Make transformations explicit  
- Optimise only when scale or cost demands it  

**TA signal:**  
Complexity added “just in case” rather than to solve a real problem.

</TabItem>

<TabItem value="separation" label="Separation of Concerns">

### Separation of Concerns
Different responsibilities should be isolated.

- Logic vs data  
- Ingestion vs transformation  
- Control plane vs data plane  

**Why it matters:**  
- Enables independent evolution  
- Improves ownership clarity  
- Reduces blast radius  

</TabItem>
</Tabs>

## Software Engineering Principles

<Tabs>
<TabItem value="solid" label="SOLID">

### SOLID Principles
Guidelines for building software that is resilient to change.

- **SRP** – One reason to change  
- **OCP** – Open for extension, closed for modification  
- **LSP** – Subtypes honour base type contracts  
- **ISP** – Prefer small, focused interfaces  
- **DIP** – Depend on abstractions, not implementations  

**TA signal:**  
SOLID violations often show up as brittle systems or painful change.

</TabItem>

<TabItem value="oop" label="4 Pillars of OOP">

### Object‑Oriented Principles

- **Inheritance** – Reuse behaviour via parent–child relationships  
- **Encapsulation** – Hide internal state; expose controlled interfaces  
- **Abstraction** – Expose essentials, hide details  
- **Polymorphism** – Treat different implementations uniformly  

**TA caution:**  
Inheritance increases coupling — composition is often safer.

</TabItem>

<TabItem value="fp" label="Functional Programming Principles">

### Functional Principles

- **Function Composition** – Build behaviour from small, reusable functions  
- **Immutability** – Avoid shared mutable state  
- **Pure Functions** – Predictable, testable behaviour  
- **Higher‑Order Functions** – Functions as first‑class values  

**TA signal:**  
Common in data processing, concurrency‑heavy systems, and pipelines.

</TabItem>
</Tabs>

## Data Engineering Principles

<Tabs>
<TabItem value="acid" label="ACID">

### ACID Principles
Transactional consistency guarantees.

- **Atomicity** – All or nothing  
- **Consistency** – Valid state transitions  
- **Isolation** – No interference between transactions  
- **Durability** – Results persist after commit  

**Typical use:** OLTP systems  
**Trade-off:** Scalability and latency

</TabItem>

<TabItem value="base" label="BASE">

### BASE Principles
Distributed system consistency model.

- **Basically Available** – System remains available under failure  
- **Soft State** – State may change without new input  
- **Eventually Consistent** – Replicas converge over time  

**Typical use:** Distributed data stores  
**Trade-off:** Immediate consistency

</TabItem>

<TabItem value="idempotency" label="Idempotency">

### Idempotency
Operations can be safely retried.

- Critical for distributed pipelines  
- Supports at‑least‑once delivery  
- Reduces failure impact  

</TabItem>

<TabItem value="schema" label="Schema Management">

### Schema-on-Write vs Schema-on-Read

- **Schema-on-write** → Enforced upfront; strong governance  
- **Schema-on-read** → Applied at consumption; flexible  

**TA focus:**  
Governance vs agility trade-offs.

</TabItem>

<TabItem value="lineage" label="Lineage & Immutability">

### Lineage & Immutability

- End‑to‑end traceability  
- Immutable raw data for auditability  
- Deterministic, documented transformations  

**TA signal:**  
Essential for debugging, compliance, and trust.

</TabItem>
</Tabs>
